यह पृष्ठ उन डेवलपर्स के लिए है जो रिपॉजिटरी क्लोन करना, इमेज को स्थानीय रूप से बिल्ड करना, या Dockerfile को कस्टमाइज़ करना चाहते हैं। यदि आप केवल पूर्व-निर्मित कंटेनर का उपयोग करना चाहते हैं, तो [शुरुआत करें](./getting-started) देखें।

## रिपॉजिटरी क्लोन करें

```bash
git clone https://github.com/f5xc-salesdemos/devcontainer.git
cd devcontainer
```

## प्रोजेक्ट संरचना

```text
.
├── Dockerfile                  # Multi-stage image build
├── docker-compose.yml          # Base compose — pulls pre-built image
├── docker-compose.build.yml    # Build override — use explicitly with -f
├── .devcontainer/              # VS Code Dev Container config
│   └── devcontainer.json
├── entrypoint.sh               # Container startup script
├── .env.example                # Example environment variables
├── docs/                       # Documentation site (Starlight)
├── claude-config/              # Claude Code configuration files
├── codex-config/               # Codex configuration
├── pi-config/                  # Pi configuration
└── omp-config/                 # Oh-My-Pi (omp) configuration
```

## बिल्ड फ़ाइल

`docker-compose.build.yml` फ़ाइल स्थानीय बिल्ड सपोर्ट जोड़ती है। यह **स्वचालित रूप से मर्ज नहीं** होती — आपको इसे `-f` फ्लैग के साथ स्पष्ट रूप से पास करना होगा। इसका मतलब है कि `docker compose up -d` (या `podman-compose up -d`) हमेशा GHCR से पूर्व-निर्मित इमेज पुल करता है, चाहे आपने रिपॉजिटरी क्लोन की हो या नहीं।

<Tabs syncKey="runtime">
<TabItem label="Docker">

आप सत्यापित कर सकते हैं कि साधारण `docker compose up` केवल पूर्व-निर्मित इमेज का उपयोग करता है (कोई `build:` कॉन्टेक्स्ट नहीं):

```bash
docker compose config
```

स्पष्ट बिल्ड कॉन्फ़िगरेशन से तुलना करें:

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml config
```

```bash
podman-compose config
```

स्पष्ट बिल्ड कॉन्फ़िगरेशन से तुलना करें:

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml config
```

</TabItem>

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash

```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>

यह बिल्ड फ़ाइल को बेस compose फ़ाइल के साथ मर्ज करता है, स्थानीय `Dockerfile` से इमेज बिल्ड करता है, और कंटेनर शुरू करता है। `--build` फ्लैग कैश्ड इमेज मौजूद होने पर भी रीबिल्ड को बाध्य करता है।

## दो-चरणीय बिल्ड आर्किटेक्चर

## टूल्स जोड़ना

`Dockerfile` को संपादित करें और अपने टूल को उचित अनुभाग में जोड़ें। जोड़ने के बाद रीबिल्ड करें:

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
</Tabs>

या VS Code में: **Dev Containers &rarr; Rebuild Container**।

### APT पैकेज

अनुभाग 2 के `apt-get install` ब्लॉक में जोड़ें:

```dockerfile
RUN apt-get update && apt-get install -y --no-install-recommends \
    your-package-here \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*
```

### npm टूल्स

अनुभाग 11 के `npm install -g` ब्लॉक में जोड़ें:

```dockerfile
RUN npm install -g \
    your-npm-tool
```

### pip टूल्स

अनुभाग 12 के `pip install` ब्लॉक में जोड़ें:

```dockerfile
RUN pip install --no-cache-dir --break-system-packages \
    your-python-tool
```

### बाइनरी डाउनलोड

अनुभाग 10 में जोड़ें या आर्किटेक्चर डिटेक्शन के साथ एक नई `RUN` लेयर बनाएं:

```dockerfile
RUN ARCH=$(dpkg --print-architecture) \
    && curl -fsSL "https://example.com/tool-linux-${ARCH}.tar.gz" \
       | tar -xz -C /usr/local/bin tool
```

## बिल्ड कैश

CI **रजिस्ट्री-आधारित कैशिंग** का उपयोग करता है — बिल्ड लेयर्स GitHub Actions कैश के बजाय GHCR (`ghcr.io/f5xc-salesdemos/devcontainer:cache-*`) में स्टोर की जाती हैं। यह 10 GB GHA कैश सीमा से बचाता है जो इस ~6 GB मल्टी-आर्किटेक्चर इमेज के लिए बार-बार पूर्ण रीबिल्ड का कारण बनती थी।

दो-चरणीय आर्किटेक्चर रजिस्ट्री कैश के साथ इस प्रकार काम करता है:

- **टूल वर्शन बम्प्स** (npm, pip, बाइनरी टूल्स) — केवल `final` चरण रीबिल्ड होता है (~5 मिनट)
- **कॉन्फ़िग फ़ाइल परिवर्तन** (`claude-config/`, `entrypoint.sh`) — केवल अंतिम कुछ लेयर्स रीबिल्ड होती हैं (~1 मिनट)
- **आधार परिवर्तन** (APT पैकेज, Go/Rust/Java वर्शन) — दोनों चरणों का पूर्ण रीबिल्ड (~30 मिनट)
