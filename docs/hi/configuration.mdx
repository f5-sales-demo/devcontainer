सारा कॉन्फ़िगरेशन प्रोजेक्ट रूट में `.env` फ़ाइल में होता है। यह फ़ाइल gitignored है — यह कभी कमिट नहीं होती।

## एन्वायरनमेंट वेरिएबल्स

### AI प्रोवाइडर

कंटेनर दो **परस्पर अनन्य** प्रमाणीकरण मोड का समर्थन करता है (प्राथमिकता क्रम में):

- **LiteLLM प्रॉक्सी** — LiteLLM प्रॉक्सी के माध्यम से रूट करने के लिए `LITELLM_BASE_URL` और `LITELLM_API_KEY` सेट करें (नीचे देखें)।
- **OAuth मोड** — केवल `CLAUDE_CODE_OAUTH_TOKEN` सेट करें (नीचे देखें)।

मोड्स को मिलाएं नहीं — बिल्कुल एक का उपयोग करें।

### Claude Max (OAuth)

| वेरिएबल | विवरण | उदाहरण |
|----------|-------------|---------|
| `CLAUDE_CODE_OAUTH_TOKEN` | Claude Max सब्सक्रिप्शन से OAuth टोकन | `sk-ant-oat01-...` |

जब सेट किया जाता है, तो Claude Code OAuth का उपयोग करके सीधे Anthropic के साथ प्रमाणित होता है — किसी API कुंजी या प्रॉक्सी की आवश्यकता नहीं। अपना टोकन Claude Code सेटिंग्स या [Anthropic कंसोल](https://console.anthropic.com/) से प्राप्त करें।

### LiteLLM प्रॉक्सी

यदि आपका LiteLLM (या अन्य) प्रॉक्सी पहले से नेटिव Anthropic Messages API — `web_search`, स्ट्रीमिंग, और टूल उपयोग सहित — बोलता है, तो `LITELLM_BASE_URL` को प्रॉक्सी डोमेन पर सेट करें। कंटेनर स्वचालित रूप से प्रोवाइडर-विशिष्ट URL प्राप्त करता है:

- Anthropic एंडपॉइंट → `${LITELLM_BASE_URL}/anthropic` (Claude Code द्वारा उपयोग किया जाता है)
- OpenAI-संगत एंडपॉइंट → `${LITELLM_BASE_URL}/openai/v1`

| वेरिएबल | विवरण | उदाहरण |
|----------|-------------|---------|
| `LITELLM_BASE_URL` | आपके LiteLLM प्रॉक्सी का डोमेन (कोई पथ प्रत्यय नहीं) | `https://litellm.example.com` |
| `LITELLM_API_KEY` | प्रॉक्सी प्रमाणीकरण के लिए API कुंजी | `your-api-key` या डिफ़ॉल्ट `litellm-proxy` |

उपयोगकर्ताओं को केवल डोमेन सेट करने की आवश्यकता है — प्रोवाइडर पथ प्रत्यय स्वचालित रूप से जोड़े जाते हैं।
रनटाइम पर, एंट्रीपॉइंट `LITELLM_API_KEY` से `ANTHROPIC_API_KEY` प्राप्त करता है ताकि Claude Code अलग उपयोगकर्ता-सामने वेरिएबल की आवश्यकता के बिना प्रॉक्सी क्रेडेंशियल का उपयोग कर सके।

इस मोड में मॉडल नाम **रीमैप नहीं** किए जाते। Claude Code अपने मानक मॉडल पहचानकर्ता (जैसे `claude-sonnet-4-6`) भेजता है और LiteLLM उन्हें आपके कॉन्फ़िगर किए गए बैकएंड पर रूट करता है।

### TLS प्रमाणपत्र सत्यापन

डिफ़ॉल्ट रूप से, Node.js सभी HTTPS कनेक्शनों के लिए TLS प्रमाणपत्रों को मान्य करता है। यदि आपका अपस्ट्रीम API प्रोवाइडर स्व-हस्ताक्षरित प्रमाणपत्र का उपयोग करता है (आंतरिक Open WebUI या LiteLLM प्रॉक्सी के साथ आम), तो आप अपनी `.env` फ़ाइल में यह जोड़कर सत्यापन अक्षम कर सकते हैं:

```ini
NODE_TLS_REJECT_UNAUTHORIZED=0
```

यह प्रोडक्शन उपयोग के लिए **अनुशंसित नहीं** है। इसे केवल तभी सेट करें जब आप कंटेनर और आपके API एंडपॉइंट के बीच नेटवर्क पथ पर भरोसा करते हों।

### AI कोडिंग एजेंट्स

कंटेनर में Claude Code के अलावा कई AI कोडिंग एजेंट्स शामिल हैं:

| एजेंट | कमांड | प्रोवाइडर |
|-------|---------|----------|
| [Codex](https://github.com/openai/codex) | `codex` | OpenAI-संगत |
| [Pi](https://github.com/mariozechner/pi-coding-agent) | `pi` | मल्टी-प्रोवाइडर (रनटाइम पर कॉन्फ़िगर करें) |
| [Oh-My-Pi](https://github.com/can1357/oh-my-pi) | `omp` | मल्टी-प्रोवाइडर (रनटाइम पर कॉन्फ़िगर करें) |

Pi और Oh-My-Pi ग्लोबल npm पैकेज के रूप में इंस्टॉल किए गए हैं। Codex एक स्टैंडअलोन बाइनरी है जो रनटाइम पर स्वयं-अपडेट होती है।

### रिमोट डिस्प्ले (noVNC)

कंटेनर हेडेड ब्राउज़रों को देखने और उनके साथ इंटरैक्ट करने के लिए एक वर्चुअल डिस्प्ले स्टैक चलाता है। कनेक्शन निर्देशों और एन्वायरनमेंट वेरिएबल्स (`ENABLE_VNC`, `VNC_RESOLUTION`, `DISPLAY`, `NOVNC_HOST_PORT`) के लिए [रिमोट डिस्प्ले (noVNC)](./vnc) देखें।

### Chrome DevTools MCP

कंटेनर में हेडलेस ब्राउज़र ऑटोमेशन के लिए एक पूर्व-कॉन्फ़िगर [Chrome DevTools MCP](https://github.com/anthropics/chrome-devtools-mcp) सर्वर शामिल है। Claude Code बिना किसी अतिरिक्त सेटअप के वेब पेज नेविगेट कर सकता है, स्क्रीनशॉट ले सकता है, और DOM का निरीक्षण कर सकता है। विवरण के लिए [Chrome DevTools MCP](./chrome) देखें।

### Git

| वेरिएबल | विवरण | कैसे सेट करें |
|----------|-------------|------------|
| `GIT_AUTHOR_NAME` | git कमिट्स के लिए नाम | `git config user.name` |
| `GIT_AUTHOR_EMAIL` | git कमिट्स के लिए ईमेल | `git config user.email` |

यदि git आपके होस्ट पर कॉन्फ़िगर है, तो दोनों मानों को ऑटो-पॉप्युलेट करें:

```bash
echo "GIT_AUTHOR_EMAIL=$(git config user.email)" >> .env
echo "GIT_AUTHOR_NAME=\"$(git config user.name)\"" >> .env
```

### SSH

| वेरिएबल | विवरण | कैसे सेट करें |
|----------|-------------|------------|
| `SSH_PRIVATE_KEY` | Base64-एनकोडेड प्राइवेट कुंजी | `base64 < ~/.ssh/id_ed25519` |

### GitHub CLI

| वेरिएबल | विवरण | कैसे सेट करें |
|----------|-------------|------------|
| `GH_TOKEN` | `gh` CLI और HTTPS git के लिए GitHub पर्सनल एक्सेस टोकन | `gh auth token` (यदि `gh` स्थानीय रूप से इंस्टॉल है) |

जब `.env` में `GH_TOKEN` सेट होता है, तो `gh` CLI स्वचालित रूप से प्रमाणित हो जाता है — `gh auth login` चलाने की आवश्यकता नहीं। एंट्रीपॉइंट `gh auth setup-git` भी चलाता है, जो git क्रेडेंशियल हेल्पर कॉन्फ़िगर करता है ताकि HTTPS `git clone` और `git push` बिना SSH कुंजियों के काम करें।

यदि `gh` आपकी होस्ट मशीन पर इंस्टॉल और प्रमाणित है, तो टोकन सीधे निकालें:

```bash
echo "GH_TOKEN=$(gh auth token)" >> .env
```

अन्यथा, [एक पर्सनल एक्सेस टोकन बनाएं](https://github.com/settings/tokens) और इसे मैन्युअल रूप से जोड़ें। फाइन-ग्रेन्ड टोकन (अनुशंसित) और क्लासिक टोकन दोनों काम करते हैं। फाइन-ग्रेन्ड टोकन आपको विशिष्ट रिपॉजिटरी तक एक्सेस स्कोप करने देते हैं। क्लासिक टोकन को प्राइवेट रिपॉजिटरी एक्सेस के लिए `repo` स्कोप की आवश्यकता होती है।

कंटेनर के अंदर प्रमाणीकरण सत्यापित करने के लिए:

```bash
gh auth status
```

## टूल्स कैसे इंस्टॉल होते हैं

## Claude Code टूल जागरूकता

कंटेनर में बिल्ट-इन कॉन्फ़िगरेशन शामिल है जो Claude Code को टूल नामों में भ्रमित होने से रोकता है। टूल जागरूकता गहरी सुरक्षा के लिए **दो** मेमोरी स्तरों पर इंस्टॉल की गई है:

- **प्रबंधित नीति** (`/etc/claude-code/CLAUDE.md`) — Docker बिल्ड समय पर इंस्टॉल। यह Claude Code के मेमोरी पदानुक्रम में सर्वोच्च-प्राथमिकता स्तर है और हमेशा लोड होता है, तब भी जब कार्यशील निर्देशिका में एक बड़ा प्रोजेक्ट `CLAUDE.md` मौजूद हो।
- **उपयोगकर्ता मेमोरी** (`~/.claude/CLAUDE.md`) — पहले स्टार्टअप पर एंट्रीपॉइंट द्वारा सीड किया गया। एक बैकअप लेयर और उपयोगकर्ताओं के लिए एक दृश्य, कस्टमाइज़ करने योग्य प्रतिलिपि प्रदान करता है।

### मेमोरी स्तर प्राथमिकता

Claude Code कई मेमोरी स्तरों से निर्देश लोड करता है (उच्चतम से निम्नतम प्राथमिकता):

1. **प्रबंधित नीति** (`/etc/claude-code/`) — हमेशा लोड, बहिष्कृत नहीं किया जा सकता
2. **प्रोजेक्ट मेमोरी** (`./CLAUDE.md`) — प्रति प्रोजेक्ट निर्देशिका लोड
3. **उपयोगकर्ता मेमोरी** (`~/.claude/CLAUDE.md`) — सभी सत्रों के लिए वैश्विक रूप से लोड
4. **स्थानीय मेमोरी** (`./CLAUDE.local.md`) — व्यक्तिगत प्रति-प्रोजेक्ट ओवरराइड

प्रबंधित नीति स्तर के बिना, उपयोगकर्ता मेमोरी में टूल जागरूकता एक बड़े प्रोजेक्ट-स्तर `CLAUDE.md` के साथ प्रतिस्पर्धा करते समय कम प्राथमिकता पा सकती है, जिससे Claude अपने PascalCase टूल नामों की जागरूकता खो देता है। प्रबंधित स्तर गारंटी देता है कि टूल जागरूकता हमेशा उच्चतम प्राथमिकता पर लोड होती है।

### स्व-परीक्षण

कॉन्फ़िगरेशन को सत्यापित करने के लिए बिल्ट-इन स्व-परीक्षण चलाएं:

```bash
claude-self-test
```

यह जांचता है कि सभी कॉन्फ़िगरेशन फ़ाइलें स्थान पर हैं और अपेक्षित टूल संदर्भ शामिल हैं।

## टूल्स जोड़ना

इमेज में टूल्स जोड़ने के लिए, `Dockerfile` संपादित करें और स्थानीय रूप से पुनर्निर्माण करें। APT पैकेज, npm टूल्स, pip टूल्स, और बाइनरी डाउनलोड जोड़ने के निर्देशों के लिए [स्थानीय विकास — टूल्स जोड़ना](./local-development#adding-tools) देखें।

## VS Code एक्सटेंशन

एक्सटेंशन `.devcontainer/devcontainer.json` में `customizations.vscode.extensions` के अंतर्गत कॉन्फ़िगर किए गए हैं और VS Code में खोलते समय स्वचालित रूप से इंस्टॉल होते हैं।

| `home` | `/home/vscode` | शेल इतिहास, टूल कॉन्फ़िग, कैश, SSH कुंजियां |

## पोर्ट मैपिंग

| होस्ट | कंटेनर | सेवा |
|------|-----------|---------|
| `127.0.0.1:${NOVNC_HOST_PORT:-6080}` | `6080` | noVNC रिमोट डिस्प्ले (केवल localhost, `NOVNC_HOST_PORT` से ओवरराइड करें) |

पोर्ट `127.0.0.1` से बाउंड हैं ताकि वे केवल आपकी मशीन से एक्सेसिबल हों, नेटवर्क से नहीं।
