Questa pagina è destinata agli sviluppatori che desiderano clonare il repository, compilare l'immagine localmente o personalizzare il Dockerfile. Se desiderate semplicemente utilizzare il container pre-compilato, consultate [Primi Passi](./getting-started).

## Clonare il Repository

```bash
git clone https://github.com/f5xc-salesdemos/devcontainer.git
cd devcontainer
```

## Struttura del Progetto

```text
.
├── Dockerfile                  # Multi-stage image build
├── docker-compose.yml          # Base compose — pulls pre-built image
├── docker-compose.build.yml    # Build override — use explicitly with -f
├── .devcontainer/              # VS Code Dev Container config
│   └── devcontainer.json
├── entrypoint.sh               # Container startup script
├── .env.example                # Example environment variables
├── docs/                       # Documentation site (Starlight)
├── claude-config/              # Claude Code configuration files
├── codex-config/               # Codex configuration
├── pi-config/                  # Pi configuration
└── omp-config/                 # Oh-My-Pi (omp) configuration
```

## File di Build

Il file `docker-compose.build.yml` aggiunge il supporto alla build locale. **Non** viene unito automaticamente — è necessario passarlo esplicitamente con i flag `-f`. Ciò significa che `docker compose up -d` (o `podman-compose up -d`) scarica sempre l'immagine pre-compilata da GHCR, indipendentemente dal fatto che abbiate clonato il repository o meno.

## Compilazione Locale

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
</Tabs>

Questo unisce il file di build con il file compose di base, compila l'immagine dal `Dockerfile` locale e avvia il container. Il flag `--build` forza una ricompilazione anche se esiste un'immagine nella cache.

## Architettura di Build a Due Stadi

Il Dockerfile utilizza una build a due stadi ottimizzata per il caching dei layer Docker:

### Stadio 1: `deps` (fondamenta stabili, ~4,5 GB)

Questi layer vengono ricompilati solo quando viene aggiornato un ARG di versione o cambiano i pacchetti APT. In caso di modifiche ai soli strumenti, BuildKit salta interamente questo stadio.

## Aggiungere Strumenti

Modificate il `Dockerfile` e aggiungete il vostro strumento nella sezione appropriata. Ricompilate dopo l'aggiunta:

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
</Tabs>

Oppure in VS Code: **Dev Containers &rarr; Ricompila Container**.

### Pacchetti APT

Aggiungete al blocco `apt-get install` della sezione 2:

```dockerfile
RUN apt-get update && apt-get install -y --no-install-recommends \
    your-package-here \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*
```

### Strumenti npm

Aggiungete al blocco `npm install -g` della sezione 11:

```dockerfile
RUN npm install -g \
    your-npm-tool
```

### Strumenti pip

Aggiungete al blocco `pip install` della sezione 12:

```dockerfile
RUN pip install --no-cache-dir --break-system-packages \
    your-python-tool
```

### Download binari

Aggiungete alla sezione 10 o create un nuovo layer `RUN` con rilevamento dell'architettura:

```dockerfile
RUN ARCH=$(dpkg --print-architecture) \
    && curl -fsSL "https://example.com/tool-linux-${ARCH}.tar.gz" \
       | tar -xz -C /usr/local/bin tool
```

## Cache di Build

La CI utilizza il **caching basato su registry** — i layer di build sono memorizzati in GHCR (`ghcr.io/f5xc-salesdemos/devcontainer:cache-*`) invece che nella cache di GitHub Actions. Questo evita il limite di 10 GB della cache GHA che causava frequenti ricompilazioni complete per questa immagine multi-architettura di ~6 GB.

L'architettura a due stadi funziona con la cache del registry in modo che:

- **Aggiornamenti di versione degli strumenti** (npm, pip, strumenti binari) — ricompilazione solo dello stadio `final` (~5 min)
- **Modifiche ai file di configurazione** (`claude-config/`, `entrypoint.sh`) — ricompilazione solo degli ultimi layer (~1 min)
- **Modifiche alle fondamenta** (pacchetti APT, versioni Go/Rust/Java) — ricompilazione completa di entrambi gli stadi (~30 min)
