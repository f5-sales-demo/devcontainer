جميع الإعدادات موجودة في ملف `.env` في جذر المشروع. هذا الملف مُتجاهَل بواسطة git — ولا يتم إرساله أبداً.

## متغيرات البيئة

### مزوّد الذكاء الاصطناعي

تدعم الحاوية وضعين للمصادقة **متعارضين** (بترتيب الأولوية):

- **وكيل LiteLLM** — عيّن `LITELLM_BASE_URL` و `LITELLM_API_KEY` للتوجيه عبر وكيل LiteLLM (انظر أدناه).
- **وضع OAuth** — عيّن `CLAUDE_CODE_OAUTH_TOKEN` فقط (انظر أدناه).

لا تجمع بين الوضعين — استخدم واحداً فقط.

### Claude Max (OAuth)

| المتغير | الوصف | مثال |
|----------|-------------|---------|
| `CLAUDE_CODE_OAUTH_TOKEN` | رمز OAuth من اشتراك Claude Max | `sk-ant-oat01-...` |

عند تعيينه، يتصل Claude Code مباشرة مع Anthropic باستخدام OAuth — لا حاجة لمفتاح API أو وكيل. احصل على رمزك من إعدادات Claude Code أو [لوحة تحكم Anthropic](https://console.anthropic.com/).

### وكيل LiteLLM

إذا كان وكيل LiteLLM (أو غيره) يتحدث بالفعل واجهة Anthropic Messages API الأصلية — بما في ذلك `web_search` والبث واستخدام الأدوات — عيّن `LITELLM_BASE_URL` إلى نطاق الوكيل. تستنتج الحاوية تلقائياً عناوين URL الخاصة بالمزوّد:

- نقطة نهاية Anthropic → `${LITELLM_BASE_URL}/anthropic` (يستخدمها Claude Code)
- نقطة نهاية متوافقة مع OpenAI → `${LITELLM_BASE_URL}/openai/v1`

| المتغير | الوصف | مثال |
|----------|-------------|---------|
| `LITELLM_BASE_URL` | نطاق وكيل LiteLLM (بدون لاحقة مسار) | `https://litellm.example.com` |
| `LITELLM_API_KEY` | مفتاح API لمصادقة الوكيل | `your-api-key` أو القيمة الافتراضية `litellm-proxy` |

يحتاج المستخدمون فقط إلى تعيين النطاق — تُضاف لاحقات مسار المزوّد تلقائياً.
في وقت التشغيل، يستنتج نقطة الدخول `ANTHROPIC_API_KEY` من `LITELLM_API_KEY` حتى يتمكن Claude Code من استهلاك بيانات اعتماد الوكيل دون الحاجة إلى متغير منفصل يواجه المستخدم.

لا يتم **إعادة تعيين** أسماء النماذج في هذا الوضع. يرسل Claude Code معرّفات النماذج القياسية (مثل `claude-sonnet-4-6`) ويوجهها LiteLLM إلى الواجهة الخلفية المُعدّة لديك.

### التحقق من شهادة TLS

بشكل افتراضي، يتحقق Node.js من شهادات TLS لجميع اتصالات HTTPS. إذا كان مزوّد API الخاص بك يستخدم شهادة موقعة ذاتياً (شائع مع وكلاء Open WebUI أو LiteLLM الداخليين)، يمكنك تعطيل التحقق بإضافة هذا إلى ملف `.env`:

```ini
NODE_TLS_REJECT_UNAUTHORIZED=0
```

هذا **غير مُوصى به** للاستخدام في بيئة الإنتاج. عيّن هذا فقط عندما تثق بمسار الشبكة بين الحاوية ونقطة نهاية API الخاصة بك.

### وكلاء البرمجة بالذكاء الاصطناعي

تتضمن الحاوية عدة وكلاء برمجة بالذكاء الاصطناعي بالإضافة إلى Claude Code:

| الوكيل | الأمر | المزوّد |
|-------|---------|----------|
| [Codex](https://github.com/openai/codex) | `codex` | متوافق مع OpenAI |
| [Pi](https://github.com/mariozechner/pi-coding-agent) | `pi` | متعدد المزوّدين (يُعدّ في وقت التشغيل) |
| [Oh-My-Pi](https://github.com/can1357/oh-my-pi) | `omp` | متعدد المزوّدين (يُعدّ في وقت التشغيل) |

Pi و Oh-My-Pi مثبتان كحزم npm عامة. Codex هو ملف ثنائي مستقل يُحدّث نفسه تلقائياً في وقت التشغيل.

### العرض عن بُعد (noVNC)

تشغّل الحاوية مجموعة عرض افتراضية لمشاهدة المتصفحات ذات الواجهة الرسومية والتفاعل معها. انظر [العرض عن بُعد (noVNC)](./vnc) للحصول على تعليمات الاتصال ومتغيرات البيئة (`ENABLE_VNC`، `VNC_RESOLUTION`، `DISPLAY`، `NOVNC_HOST_PORT`).

### Chrome DevTools MCP

تتضمن الحاوية خادم [Chrome DevTools MCP](https://github.com/anthropics/chrome-devtools-mcp) مُعدّ مسبقاً لأتمتة المتصفح بدون واجهة رسومية. يمكن لـ Claude Code التنقل في صفحات الويب والتقاط لقطات شاشة وفحص DOM دون أي إعداد إضافي. انظر [Chrome DevTools MCP](./chrome) للتفاصيل.

### Git

| المتغير | الوصف | كيفية التعيين |
|----------|-------------|------------|
| `GIT_AUTHOR_NAME` | الاسم لعمليات git commit | `git config user.name` |
| `GIT_AUTHOR_EMAIL` | البريد الإلكتروني لعمليات git commit | `git config user.email` |

إذا كان git مُعدّاً على جهازك المضيف، يمكنك ملء القيمتين تلقائياً:

```bash
echo "GIT_AUTHOR_EMAIL=$(git config user.email)" >> .env
echo "GIT_AUTHOR_NAME=\"$(git config user.name)\"" >> .env
```

### SSH

| المتغير | الوصف | كيفية التعيين |
|----------|-------------|------------|
| `SSH_PRIVATE_KEY` | المفتاح الخاص مُرمّز بـ Base64 | `base64 < ~/.ssh/id_ed25519` |

### GitHub CLI

| المتغير | الوصف | كيفية التعيين |
|----------|-------------|------------|
| `GH_TOKEN` | رمز الوصول الشخصي لـ GitHub لأداة `gh` CLI و git عبر HTTPS | `gh auth token` (إذا كان `gh` مثبتاً محلياً) |

عند تعيين `GH_TOKEN` في `.env`، تتم مصادقة أداة `gh` CLI تلقائياً — لا حاجة لتشغيل `gh auth login`. تقوم نقطة الدخول أيضاً بتشغيل `gh auth setup-git`، الذي يُعدّ مساعد بيانات اعتماد git بحيث يعمل `git clone` و `git push` عبر HTTPS بدون مفاتيح SSH.

إذا كان `gh` مثبتاً ومُصادقاً على جهازك المضيف، استخرج الرمز مباشرة:

```bash
echo "GH_TOKEN=$(gh auth token)" >> .env
```

وإلا، [أنشئ رمز وصول شخصي](https://github.com/settings/tokens) وأضفه يدوياً. تعمل كل من الرموز الدقيقة (مُوصى بها) والرموز الكلاسيكية. تتيح لك الرموز الدقيقة تحديد الوصول لمستودعات معينة. تتطلب الرموز الكلاسيكية نطاق `repo` للوصول إلى المستودعات الخاصة.

للتحقق من المصادقة داخل الحاوية:

```bash
gh auth status
```

## كيفية تثبيت الأدوات

يتم تثبيت كل بيئة تشغيل لغة وأداة CLI وأداة مباشرة في `Dockerfile` في وقت بناء الصورة. تُنشر الصورة المبنية مسبقاً إلى `ghcr.io/f5xc-salesdemos/devcontainer:latest` مع كل دفع إلى `main`، لذا لا يحتاج معظم المستخدمين إلى البناء محلياً.

## وعي Claude Code بالأدوات

تتضمن الحاوية إعدادات مُدمجة تمنع Claude Code من الخلط بين أسماء الأدوات. يتم تثبيت وعي الأدوات في **مستويين** من الذاكرة كدفاع متعدد الطبقات:

- **السياسة المُدارة** (`/etc/claude-code/CLAUDE.md`) — تُثبّت في وقت بناء Docker. هذا هو المستوى الأعلى أولوية في تسلسل ذاكرة Claude Code ويتم تحميله دائماً، حتى عند وجود ملف `CLAUDE.md` كبير للمشروع في مجلد العمل.
- **ذاكرة المستخدم** (`~/.claude/CLAUDE.md`) — تُزرع بواسطة نقطة الدخول عند أول تشغيل. توفر طبقة احتياطية ونسخة مرئية وقابلة للتخصيص للمستخدمين.

### أولوية مستويات الذاكرة

يحمّل Claude Code التعليمات من مستويات ذاكرة متعددة (من الأعلى إلى الأدنى أولوية):

1. **السياسة المُدارة** (`/etc/claude-code/`) — تُحمّل دائماً، ولا يمكن استبعادها
2. **ذاكرة المشروع** (`./CLAUDE.md`) — تُحمّل حسب مجلد المشروع
3. **ذاكرة المستخدم** (`~/.claude/CLAUDE.md`) — تُحمّل عالمياً لجميع الجلسات
4. **الذاكرة المحلية** (`./CLAUDE.local.md`) — تجاوزات شخصية لكل مشروع

بدون مستوى السياسة المُدارة، يمكن تخفيض أولوية وعي الأدوات في ذاكرة المستخدم عند التنافس مع ملف `CLAUDE.md` كبير على مستوى المشروع، مما يتسبب في فقدان Claude لوعيه بأسماء أدواته المكتوبة بنمط PascalCase. يضمن المستوى المُدار تحميل وعي الأدوات دائماً بأعلى أولوية.

### الاختبار الذاتي

شغّل الاختبار الذاتي المُدمج للتحقق من الإعدادات:

```bash
claude-self-test
```

يتحقق هذا من أن جميع ملفات الإعدادات في مكانها وتحتوي على مراجع الأدوات المتوقعة.

## إضافة الأدوات

لإضافة أدوات إلى الصورة، عدّل `Dockerfile` وأعد البناء محلياً. انظر [التطوير المحلي — إضافة الأدوات](./local-development#adding-tools) للحصول على تعليمات حول إضافة حزم APT وأدوات npm وأدوات pip والتنزيلات الثنائية.

## إضافات VS Code

تُعدّ الإضافات في `.devcontainer/devcontainer.json` تحت `customizations.vscode.extensions` وتُثبّت تلقائياً عند الفتح في VS Code.

## استمرارية البيانات

جميع البيانات موجودة في وحدات تخزين مُسمّاة — لا يتم تركيب أي مجلدات من المضيف:

| وحدة التخزين | نقطة التركيب | المحتويات |
|--------|-------------|----------|
| `workspace` | `/workspace` | مستودعاتك المُستنسخة وملفات المشروع |
| `home` | `/home/vscode` | سجل الأوامر، إعدادات الأدوات، الذاكرة المؤقتة، مفاتيح SSH |

## تعيين المنافذ

| المضيف | الحاوية | الخدمة |
|------|-----------|---------|
| `127.0.0.1:${NOVNC_HOST_PORT:-6080}` | `6080` | عرض noVNC عن بُعد (localhost فقط، يمكن التجاوز بـ `NOVNC_HOST_PORT`) |

المنافذ مربوطة بـ `127.0.0.1` بحيث تكون متاحة فقط من جهازك، وليس من الشبكة.
