هذه الصفحة مخصصة للمطورين الذين يرغبون في استنساخ المستودع أو بناء الصورة محلياً أو تخصيص ملف Dockerfile. إذا كنت تريد فقط استخدام الحاوية المبنية مسبقاً، راجع [البدء](./getting-started).

## استنساخ المستودع

```bash
git clone https://github.com/f5xc-salesdemos/devcontainer.git
cd devcontainer
```

## هيكل المشروع

```text
.
├── Dockerfile                  # Multi-stage image build
├── docker-compose.yml          # Base compose — pulls pre-built image
├── docker-compose.build.yml    # Build override — use explicitly with -f
├── .devcontainer/              # VS Code Dev Container config
│   └── devcontainer.json
├── entrypoint.sh               # Container startup script
├── .env.example                # Example environment variables
├── docs/                       # Documentation site (Starlight)
├── claude-config/              # Claude Code configuration files
├── codex-config/               # Codex configuration
├── pi-config/                  # Pi configuration
└── omp-config/                 # Oh-My-Pi (omp) configuration
```

## ملف البناء

يضيف ملف `docker-compose.build.yml` دعم البناء المحلي. وهو **لا** يُدمج تلقائياً — يجب تمريره صراحةً باستخدام علامات `-f`. هذا يعني أن `docker compose up -d` (أو `podman-compose up -d`) يسحب دائماً الصورة المبنية مسبقاً من GHCR، سواء قمت باستنساخ المستودع أم لا.


قارن مع تكوين البناء الصريح:

يدمج هذا ملف البناء مع ملف compose الأساسي، ويبني الصورة من ملف `Dockerfile` المحلي، ويشغّل الحاوية. تفرض علامة `--build` إعادة البناء حتى لو كانت هناك صورة مخزنة مؤقتاً.

## بنية البناء ثنائية المراحل

## إضافة أدوات

حرّر ملف `Dockerfile` وأضف أداتك إلى القسم المناسب. أعد البناء بعد الإضافة:

<TabItem label="Docker">

```bash

```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>

أو في VS Code: **Dev Containers &rarr; Rebuild Container**.

### حزم APT

أضف إلى كتلة `apt-get install` في القسم 2:

```dockerfile
RUN apt-get update && apt-get install -y --no-install-recommends \
    your-package-here \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*
```

### أدوات npm

أضف إلى كتلة `npm install -g` في القسم 11:

```dockerfile
RUN npm install -g \
    your-npm-tool
```

### أدوات pip

أضف إلى كتلة `pip install` في القسم 12:

```dockerfile
RUN pip install --no-cache-dir --break-system-packages \
    your-python-tool
```

### تنزيلات ثنائية

أضف إلى القسم 10 أو أنشئ طبقة `RUN` جديدة مع اكتشاف البنية:

```dockerfile
RUN ARCH=$(dpkg --print-architecture) \
    && curl -fsSL "https://example.com/tool-linux-${ARCH}.tar.gz" \
       | tar -xz -C /usr/local/bin tool
```

## ذاكرة التخزين المؤقت للبناء

يستخدم التكامل المستمر **التخزين المؤقت القائم على السجل** — تُخزَّن طبقات البناء في GHCR (`ghcr.io/f5xc-salesdemos/devcontainer:cache-*`) بدلاً من ذاكرة التخزين المؤقت لـ GitHub Actions. يتجنب هذا حد ذاكرة التخزين المؤقت لـ GHA البالغ 10 غيغابايت الذي كان يسبب عمليات إعادة بناء كاملة متكررة لهذه الصورة متعددة البنيات التي يبلغ حجمها ~6 غيغابايت.

تعمل البنية ثنائية المراحل مع ذاكرة التخزين المؤقت للسجل بحيث:

- **تحديثات إصدارات الأدوات** (npm، pip، أدوات ثنائية) — تُعاد بناء مرحلة `final` فقط (~5 دقائق)
- **تغييرات ملفات التكوين** (`claude-config/`، `entrypoint.sh`) — تُعاد بناء الطبقات الأخيرة فقط (~1 دقيقة)
- **تغييرات الأسس** (حزم APT، إصدارات Go/Rust/Java) — إعادة بناء كاملة لكلتا المرحلتين (~30 دقيقة)
