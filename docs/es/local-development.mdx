Esta página está dirigida a desarrolladores que desean clonar el repositorio, compilar la imagen localmente o personalizar el Dockerfile. Si solo desea utilizar el contenedor precompilado, consulte [Primeros Pasos](./getting-started).

## Clonar el Repositorio

```bash
git clone https://github.com/f5xc-salesdemos/devcontainer.git
cd devcontainer
```

## Estructura del Proyecto

```text
.
├── Dockerfile                  # Multi-stage image build
├── docker-compose.yml          # Base compose — pulls pre-built image
├── docker-compose.build.yml    # Build override — use explicitly with -f
├── .devcontainer/              # VS Code Dev Container config
│   └── devcontainer.json
├── entrypoint.sh               # Container startup script
├── .env.example                # Example environment variables
├── docs/                       # Documentation site (Starlight)
├── claude-config/              # Claude Code configuration files
├── codex-config/               # Codex configuration
├── pi-config/                  # Pi configuration
└── omp-config/                 # Oh-My-Pi (omp) configuration
```

## Archivo de Compilación

El archivo `docker-compose.build.yml` añade soporte de compilación local. **No** se fusiona automáticamente — debe pasarlo explícitamente con las flags `-f`. Esto significa que `docker compose up -d` (o `podman-compose up -d`) siempre descarga la imagen precompilada de GHCR, independientemente de si clonó el repositorio o no.

## Compilación Local

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
</Tabs>

Esto fusiona el archivo de compilación con el archivo compose base, compila la imagen desde el `Dockerfile` local e inicia el contenedor. La flag `--build` fuerza una recompilación incluso si existe una imagen en caché.

## Arquitectura de Compilación en Dos Etapas

El Dockerfile utiliza una compilación en dos etapas optimizada para el almacenamiento en caché de capas de Docker:

### Etapa 1: `deps` (fundamentos estables, ~4.5 GB)

Estas capas solo se recompilan cuando se actualiza una versión ARG o cambian los paquetes APT. En cambios típicos que solo afectan herramientas, BuildKit omite esta etapa por completo.

## Añadir Herramientas

Edite el `Dockerfile` y añada su herramienta en la sección correspondiente. Recompile después de añadirla:

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
</Tabs>

O en VS Code: **Dev Containers &rarr; Rebuild Container**.

### Paquetes APT

Añada al bloque `apt-get install` de la sección 2:

```dockerfile
RUN apt-get update && apt-get install -y --no-install-recommends \
    your-package-here \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*
```

### Herramientas npm

Añada al bloque `npm install -g` de la sección 11:

```dockerfile
RUN npm install -g \
    your-npm-tool
```

### Herramientas pip

Añada al bloque `pip install` de la sección 12:

```dockerfile
RUN pip install --no-cache-dir --break-system-packages \
    your-python-tool
```

### Descargas binarias

Añada a la sección 10 o cree una nueva capa `RUN` con detección de arquitectura:

```dockerfile
RUN ARCH=$(dpkg --print-architecture) \
    && curl -fsSL "https://example.com/tool-linux-${ARCH}.tar.gz" \
       | tar -xz -C /usr/local/bin tool
```

## Caché de Compilación

CI utiliza **caché basado en registro** — las capas de compilación se almacenan en GHCR (`ghcr.io/f5xc-salesdemos/devcontainer:cache-*`) en lugar de la caché de GitHub Actions. Esto evita el límite de 10 GB de la caché de GHA que causaba recompilaciones completas frecuentes para esta imagen multi-arquitectura de ~6 GB.

La arquitectura en dos etapas funciona con la caché de registro de manera que:

- **Actualizaciones de versión de herramientas** (npm, pip, herramientas binarias) — solo recompila la etapa `final` (~5 min)
- **Cambios en archivos de configuración** (`claude-config/`, `entrypoint.sh`) — solo recompila las últimas capas (~1 min)
- **Cambios en los fundamentos** (paquetes APT, versiones de Go/Rust/Java) — recompilación completa de ambas etapas (~30 min)
