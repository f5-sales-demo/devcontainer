Toda la configuración reside en el archivo `.env` en la raíz del proyecto. Este archivo está en el gitignore — nunca se incluye en los commits.

## Variables de Entorno

### Proveedor de IA

El contenedor admite dos modos de autenticación **mutuamente excluyentes** (en orden de prioridad):

- **Proxy LiteLLM** — establezca `LITELLM_BASE_URL` y `LITELLM_API_KEY` para enrutar a través de un proxy LiteLLM (ver más abajo).
- **Modo OAuth** — establezca únicamente `CLAUDE_CODE_OAUTH_TOKEN` (ver más abajo).

No combine los modos — use exactamente uno.

### Claude Max (OAuth)

| Variable | Descripción | Ejemplo |
|----------|-------------|---------|
| `CLAUDE_CODE_OAUTH_TOKEN` | Token OAuth de una suscripción Claude Max | `sk-ant-oat01-...` |

Cuando está configurado, Claude Code se autentica directamente con Anthropic usando OAuth — no se necesita clave API ni proxy. Obtenga su token desde la configuración de Claude Code o la [consola de Anthropic](https://console.anthropic.com/).

### Proxy LiteLLM

Si su proxy LiteLLM (u otro) ya habla la API nativa de Mensajes de Anthropic — incluyendo `web_search`, streaming y uso de herramientas — establezca `LITELLM_BASE_URL` con el dominio del proxy. El contenedor deriva automáticamente las URLs específicas del proveedor:

- Endpoint de Anthropic → `${LITELLM_BASE_URL}/anthropic` (usado por Claude Code)
- Endpoint compatible con OpenAI → `${LITELLM_BASE_URL}/openai/v1`

| Variable | Descripción | Ejemplo |
|----------|-------------|---------|
| `LITELLM_BASE_URL` | Dominio de su proxy LiteLLM (sin sufijo de ruta) | `https://litellm.example.com` |
| `LITELLM_API_KEY` | Clave API para autenticación del proxy | `your-api-key` o por defecto `litellm-proxy` |

Los usuarios solo necesitan establecer el dominio — los sufijos de ruta del proveedor se agregan automáticamente.
En tiempo de ejecución, el entrypoint deriva `ANTHROPIC_API_KEY` de `LITELLM_API_KEY` para que Claude Code pueda consumir la credencial del proxy sin requerir una variable adicional orientada al usuario.

Los nombres de modelos **no** se remapean en este modo. Claude Code envía sus identificadores de modelo estándar (p. ej. `claude-sonnet-4-6`) y LiteLLM los enruta a su backend configurado.

### Validación de Certificados TLS

Por defecto, Node.js valida los certificados TLS para todas las conexiones HTTPS. Si su proveedor de API upstream usa un certificado autofirmado (común con proxies internos de Open WebUI o LiteLLM), puede deshabilitar la validación agregando esto a su archivo `.env`:

```ini
NODE_TLS_REJECT_UNAUTHORIZED=0
```

Esto **no se recomienda** para uso en producción. Solo establezca esto cuando confíe en la ruta de red entre el contenedor y su endpoint de API.

### Agentes de Codificación IA

El contenedor incluye varios agentes de codificación IA además de Claude Code:

| Agente | Comando | Proveedor |
|--------|---------|-----------|
| [Codex](https://github.com/openai/codex) | `codex` | Compatible con OpenAI |
| [Pi](https://github.com/mariozechner/pi-coding-agent) | `pi` | Multi-proveedor (configurar en tiempo de ejecución) |
| [Oh-My-Pi](https://github.com/can1357/oh-my-pi) | `omp` | Multi-proveedor (configurar en tiempo de ejecución) |

Pi y Oh-My-Pi se instalan como paquetes npm globales. Codex es un binario independiente que se auto-actualiza en tiempo de ejecución.

### Pantalla Remota (noVNC)

El contenedor ejecuta una pila de pantalla virtual para observar e interactuar con navegadores con interfaz gráfica. Consulte [Pantalla Remota (noVNC)](./vnc) para instrucciones de conexión y variables de entorno (`ENABLE_VNC`, `VNC_RESOLUTION`, `DISPLAY`, `NOVNC_HOST_PORT`).

### Chrome DevTools MCP

El contenedor incluye un servidor [Chrome DevTools MCP](https://github.com/anthropics/chrome-devtools-mcp) preconfigurado para automatización de navegador headless. Claude Code puede navegar páginas web, tomar capturas de pantalla e inspeccionar el DOM sin ninguna configuración adicional. Consulte [Chrome DevTools MCP](./chrome) para más detalles.

### Git

| Variable | Descripción | Cómo Configurar |
|----------|-------------|-----------------|
| `GIT_AUTHOR_NAME` | Nombre para los commits de git | `git config user.name` |
| `GIT_AUTHOR_EMAIL` | Email para los commits de git | `git config user.email` |

Si git está configurado en su host, auto-complete ambos valores:

```bash
echo "GIT_AUTHOR_EMAIL=$(git config user.email)" >> .env
echo "GIT_AUTHOR_NAME=\"$(git config user.name)\"" >> .env
```

### SSH

| Variable | Descripción | Cómo Configurar |
|----------|-------------|-----------------|
| `SSH_PRIVATE_KEY` | Clave privada codificada en Base64 | `base64 < ~/.ssh/id_ed25519` |

### GitHub CLI

| Variable | Descripción | Cómo Configurar |
|----------|-------------|-----------------|
| `GH_TOKEN` | Token de acceso personal de GitHub para `gh` CLI y git HTTPS | `gh auth token` (si `gh` está instalado localmente) |

Cuando `GH_TOKEN` está configurado en `.env`, la CLI `gh` se autentica automáticamente — no es necesario ejecutar `gh auth login`. El entrypoint también ejecuta `gh auth setup-git`, que configura el asistente de credenciales de git para que `git clone` y `git push` por HTTPS funcionen sin claves SSH.

Si `gh` está instalado y autenticado en su máquina host, extraiga el token directamente:

```bash
echo "GH_TOKEN=$(gh auth token)" >> .env
```

De lo contrario, [cree un token de acceso personal](https://github.com/settings/tokens) y agréguelo manualmente. Tanto los tokens de granularidad fina (recomendados) como los tokens clásicos funcionan. Los tokens de granularidad fina le permiten limitar el acceso a repositorios específicos. Los tokens clásicos requieren el alcance `repo` para acceso a repositorios privados.

Para verificar la autenticación dentro del contenedor:

```bash
gh auth status
```

## Cómo Se Instalan las Herramientas

Cada runtime de lenguaje, CLI y herramienta se instala directamente en el `Dockerfile` en tiempo de compilación de la imagen. La imagen precompilada se publica en `ghcr.io/f5xc-salesdemos/devcontainer:latest` con cada push a `main`, por lo que la mayoría de los usuarios nunca necesitan compilar localmente.

## Conocimiento de Herramientas de Claude Code

El contenedor incluye configuración integrada que evita que Claude Code confunda los nombres de las herramientas. El conocimiento de herramientas se instala en **dos** niveles de memoria para defensa en profundidad:

- **Política administrada** (`/etc/claude-code/CLAUDE.md`) — instalada en tiempo de compilación de Docker. Este es el nivel de mayor prioridad en la jerarquía de memoria de Claude Code y siempre se carga, incluso cuando existe un `CLAUDE.md` de proyecto grande en el directorio de trabajo.
- **Memoria de usuario** (`~/.claude/CLAUDE.md`) — sembrada por el entrypoint en el primer inicio. Proporciona una capa de respaldo y una copia visible y personalizable para los usuarios.

### Prioridad de Niveles de Memoria

Claude Code carga instrucciones desde múltiples niveles de memoria (de mayor a menor prioridad):

1. **Política administrada** (`/etc/claude-code/`) — Siempre se carga, no puede excluirse
2. **Memoria de proyecto** (`./CLAUDE.md`) — Se carga por directorio de proyecto
3. **Memoria de usuario** (`~/.claude/CLAUDE.md`) — Se carga globalmente para todas las sesiones
4. **Memoria local** (`./CLAUDE.local.md`) — Sobrescrituras personales por proyecto

Sin el nivel de Política administrada, el conocimiento de herramientas en la Memoria de usuario puede ser desprioritizado cuando compite con un `CLAUDE.md` grande a nivel de proyecto, causando que Claude pierda el conocimiento de sus nombres de herramientas en PascalCase. El nivel Administrado garantiza que el conocimiento de herramientas siempre se cargue con la mayor prioridad.

### Auto-Test

Ejecute el auto-test integrado para verificar la configuración:

```bash
claude-self-test
```

Esto verifica que todos los archivos de configuración estén en su lugar y contengan las referencias de herramientas esperadas.

## Agregar Herramientas

Para agregar herramientas a la imagen, edite el `Dockerfile` y recompile localmente. Consulte [Desarrollo Local — Agregar Herramientas](./local-development#adding-tools) para instrucciones sobre cómo agregar paquetes APT, herramientas npm, herramientas pip y descargas de binarios.

## Extensiones de VS Code

Las extensiones se configuran en `.devcontainer/devcontainer.json` bajo `customizations.vscode.extensions` y se instalan automáticamente al abrir en VS Code.

## Persistencia de Datos

Todos los datos residen en volúmenes con nombre — no se montan directorios del host:

| Volumen | Punto de Montaje | Contenido |
|---------|-------------------|-----------|
| `workspace` | `/workspace` | Sus repositorios clonados y archivos de proyecto |
| `home` | `/home/vscode` | Historial de shell, configuraciones de herramientas, cachés, claves SSH |

## Mapeo de Puertos

| Host | Contenedor | Servicio |
|------|------------|----------|
| `127.0.0.1:${NOVNC_HOST_PORT:-6080}` | `6080` | Pantalla remota noVNC (solo localhost, sobrescriba con `NOVNC_HOST_PORT`) |

Los puertos están vinculados a `127.0.0.1` para que solo sean accesibles desde su máquina, no desde la red.
