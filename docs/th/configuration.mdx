การกำหนดค่าทั้งหมดอยู่ในไฟล์ `.env` ที่รูทของโปรเจกต์ ไฟล์นี้ถูก gitignore — จะไม่ถูก commit

## ตัวแปรสภาพแวดล้อม

### ผู้ให้บริการ AI

คอนเทนเนอร์รองรับโหมดการยืนยันตัวตนแบบ **ไม่สามารถใช้ร่วมกันได้** สองโหมด (ตามลำดับความสำคัญ):

- **LiteLLM proxy** — ตั้งค่า `LITELLM_BASE_URL` และ `LITELLM_API_KEY` เพื่อส่งผ่าน LiteLLM proxy (ดูด้านล่าง)
- **โหมด OAuth** — ตั้งค่า `CLAUDE_CODE_OAUTH_TOKEN` เท่านั้น (ดูด้านล่าง)

อย่ารวมโหมดเข้าด้วยกัน — ใช้เพียงโหมดเดียว

### Claude Max (OAuth)

| ตัวแปร | คำอธิบาย | ตัวอย่าง |
|----------|-------------|---------|
| `CLAUDE_CODE_OAUTH_TOKEN` | โทเค็น OAuth จากการสมัครสมาชิก Claude Max | `sk-ant-oat01-...` |

เมื่อตั้งค่าแล้ว Claude Code จะยืนยันตัวตนโดยตรงกับ Anthropic โดยใช้ OAuth — ไม่ต้องใช้ API key หรือ proxy ดูโทเค็นของคุณได้จากการตั้งค่า Claude Code หรือ [Anthropic console](https://console.anthropic.com/)

### LiteLLM Proxy

หาก LiteLLM (หรือ proxy อื่น) ของคุณรองรับ Anthropic Messages API แบบเนทีฟอยู่แล้ว — รวมถึง `web_search`, streaming และ tool use — ให้ตั้งค่า `LITELLM_BASE_URL` เป็นโดเมนของ proxy คอนเทนเนอร์จะสร้าง URL เฉพาะผู้ให้บริการโดยอัตโนมัติ:

- Anthropic endpoint → `${LITELLM_BASE_URL}/anthropic` (ใช้โดย Claude Code)
- OpenAI-compatible endpoint → `${LITELLM_BASE_URL}/openai/v1`

| ตัวแปร | คำอธิบาย | ตัวอย่าง |
|----------|-------------|---------|
| `LITELLM_BASE_URL` | โดเมนของ LiteLLM proxy ของคุณ (ไม่ต้องมี path suffix) | `https://litellm.example.com` |
| `LITELLM_API_KEY` | API key สำหรับการยืนยันตัวตนกับ proxy | `your-api-key` หรือค่าเริ่มต้นเป็น `litellm-proxy` |

ผู้ใช้ต้องตั้งค่าเฉพาะโดเมนเท่านั้น — path suffix ของผู้ให้บริการจะถูกเพิ่มโดยอัตโนมัติ
ในขณะรันไทม์ entrypoint จะสร้าง `ANTHROPIC_API_KEY` จาก `LITELLM_API_KEY` เพื่อให้ Claude Code สามารถใช้ข้อมูลประจำตัวของ proxy ได้โดยไม่ต้องมีตัวแปรแยกสำหรับผู้ใช้

ชื่อโมเดล **ไม่ได้** ถูก remap ในโหมดนี้ Claude Code จะส่งตัวระบุโมเดลมาตรฐาน (เช่น `claude-sonnet-4-6`) และ LiteLLM จะส่งต่อไปยังแบ็กเอนด์ที่คุณกำหนดค่าไว้

### การตรวจสอบใบรับรอง TLS

ตามค่าเริ่มต้น Node.js จะตรวจสอบใบรับรอง TLS สำหรับการเชื่อมต่อ HTTPS ทั้งหมด หากผู้ให้บริการ API ต้นทางของคุณใช้ใบรับรองแบบ self-signed (พบบ่อยกับ Open WebUI หรือ LiteLLM proxy ภายในองค์กร) คุณสามารถปิดการตรวจสอบได้โดยเพิ่มบรรทัดนี้ในไฟล์ `.env` ของคุณ:

```ini
NODE_TLS_REJECT_UNAUTHORIZED=0
```

**ไม่แนะนำ** ให้ใช้ในสภาพแวดล้อมการใช้งานจริง ตั้งค่านี้เฉพาะเมื่อคุณเชื่อถือเส้นทางเครือข่ายระหว่างคอนเทนเนอร์และ API endpoint ของคุณ

### Git

| ตัวแปร | คำอธิบาย | วิธีการตั้งค่า |
|----------|-------------|------------|
| `GIT_AUTHOR_NAME` | ชื่อสำหรับ git commits | `git config user.name` |
| `GIT_AUTHOR_EMAIL` | อีเมลสำหรับ git commits | `git config user.email` |

หาก git ถูกกำหนดค่าไว้บนเครื่องโฮสต์ของคุณ ให้ดึงค่าทั้งสองโดยอัตโนมัติ:

```bash
echo "GIT_AUTHOR_EMAIL=$(git config user.email)" >> .env
echo "GIT_AUTHOR_NAME=\"$(git config user.name)\"" >> .env
```

|----------|-------------|------------|
| `SSH_PRIVATE_KEY` | Private key ที่เข้ารหัสแบบ Base64 | `base64 < ~/.ssh/id_ed25519` |

### GitHub CLI

| ตัวแปร | คำอธิบาย | วิธีการตั้งค่า |
|----------|-------------|------------|
| `GH_TOKEN` | GitHub personal access token สำหรับ `gh` CLI และ HTTPS git | `gh auth token` (หาก `gh` ถูกติดตั้งในเครื่อง) |

เมื่อตั้งค่า `GH_TOKEN` ในไฟล์ `.env` แล้ว `gh` CLI จะยืนยันตัวตนโดยอัตโนมัติ — ไม่ต้องรัน `gh auth login` นอกจากนี้ entrypoint ยังรัน `gh auth setup-git` ซึ่งกำหนดค่า git credential helper เพื่อให้ `git clone` และ `git push` ผ่าน HTTPS ทำงานได้โดยไม่ต้องใช้ SSH keys

หาก `gh` ถูกติดตั้งและยืนยันตัวตนแล้วบนเครื่องโฮสต์ของคุณ ให้ดึงโทเค็นโดยตรง:

```bash
echo "GH_TOKEN=$(gh auth token)" >> .env
```

มิฉะนั้น [สร้าง personal access token](https://github.com/settings/tokens) และเพิ่มด้วยตนเอง ทั้ง fine-grained tokens (แนะนำ) และ classic tokens ใช้ได้ Fine-grained tokens ให้คุณจำกัดการเข้าถึงเฉพาะ repository ที่ต้องการ Classic tokens ต้องการ scope `repo` สำหรับการเข้าถึง private repository

เพื่อตรวจสอบการยืนยันตัวตนภายในคอนเทนเนอร์:

```bash
gh auth status
```

## การรับรู้เครื่องมือของ Claude Code

คอนเทนเนอร์มีการกำหนดค่าที่ฝังไว้เพื่อป้องกันไม่ให้ Claude Code สับสนชื่อเครื่องมือ การรับรู้เครื่องมือถูกติดตั้งใน **สอง** ระดับหน่วยความจำเพื่อการป้องกันแบบเชิงลึก:

- **Managed policy** (`/etc/claude-code/CLAUDE.md`) — ติดตั้งในเวลาสร้าง Docker นี่เป็นระดับที่มีความสำคัญสูงสุดในลำดับชั้นหน่วยความจำของ Claude Code และจะถูกโหลดเสมอ แม้ว่าจะมี `CLAUDE.md` ระดับโปรเจกต์ขนาดใหญ่ในไดเรกทอรีการทำงาน
- **User memory** (`~/.claude/CLAUDE.md`) — ถูกสร้างขึ้นโดย entrypoint ในการเริ่มต้นครั้งแรก ให้ชั้นสำรองและสำเนาที่มองเห็นได้และปรับแต่งได้สำหรับผู้ใช้

### ลำดับความสำคัญของระดับหน่วยความจำ

Claude Code โหลดคำสั่งจากหลายระดับหน่วยความจำ (จากสูงสุดไปต่ำสุด):

1. **Managed policy** (`/etc/claude-code/`) — โหลดเสมอ ไม่สามารถยกเว้นได้
2. **Project memory** (`./CLAUDE.md`) — โหลดตามไดเรกทอรีโปรเจกต์
3. **User memory** (`~/.claude/CLAUDE.md`) — โหลดแบบ global สำหรับทุกเซสชัน
4. **Local memory** (`./CLAUDE.local.md`) — การแทนที่ส่วนตัวระดับโปรเจกต์

หากไม่มีระดับ Managed policy การรับรู้เครื่องมือใน User memory อาจถูกลดความสำคัญเมื่อแข่งกับ `CLAUDE.md` ระดับโปรเจกต์ขนาดใหญ่ ทำให้ Claude สูญเสียการรับรู้ชื่อเครื่องมือแบบ PascalCase ระดับ Managed รับประกันว่าการรับรู้เครื่องมือจะถูกโหลดในความสำคัญสูงสุดเสมอ

### การทดสอบตัวเอง

รันการทดสอบตัวเองที่มีอยู่ในตัวเพื่อตรวจสอบการกำหนดค่า:

```bash
claude-self-test
```

การทดสอบนี้จะตรวจสอบว่าไฟล์การกำหนดค่าทั้งหมดอยู่ในที่ที่ถูกต้องและมีการอ้างอิงเครื่องมือที่คาดหวัง

## การเพิ่มเครื่องมือ

หากต้องการเพิ่มเครื่องมือลงใน image ให้แก้ไข `Dockerfile` และสร้างใหม่ในเครื่อง ดู [การพัฒนาในเครื่อง — การเพิ่มเครื่องมือ](./local-development#adding-tools) สำหรับคำแนะนำในการเพิ่ม APT packages, npm tools, pip tools และการดาวน์โหลดไบนารี

## ส่วนขยาย VS Code

ส่วนขยายถูกกำหนดค่าใน `.devcontainer/devcontainer.json` ภายใต้ `customizations.vscode.extensions` และถูกติดตั้งโดยอัตโนมัติเมื่อเปิดใน VS Code

## การคงอยู่ของข้อมูล

ข้อมูลทั้งหมดอยู่ใน named volumes — ไม่มีการ mount ไดเรกทอรีของโฮสต์:

| Volume | จุด Mount | เนื้อหา |
|--------|-------------|----------|
| `workspace` | `/workspace` | Repository ที่โคลนและไฟล์โปรเจกต์ของคุณ |
| `home` | `/home/vscode` | ประวัติ shell, การกำหนดค่าเครื่องมือ, แคช, SSH keys |

## การแมปพอร์ต

| โฮสต์ | คอนเทนเนอร์ | บริการ |
|------|-----------|---------|
| `127.0.0.1:${NOVNC_HOST_PORT:-6080}` | `6080` | noVNC remote display (localhost เท่านั้น, แทนที่ด้วย `NOVNC_HOST_PORT`) |

พอร์ตถูกผูกกับ `127.0.0.1` เพื่อให้เข้าถึงได้จากเครื่องของคุณเท่านั้น ไม่ใช่จากเครือข่าย
