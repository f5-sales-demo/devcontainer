การกำหนดค่าทั้งหมดอยู่ในไฟล์ `.env` ที่รากของโปรเจกต์ ไฟล์นี้ถูก gitignore ไว้ — จะไม่ถูก commit

## ตัวแปรสภาพแวดล้อม

### ผู้ให้บริการ AI

คอนเทนเนอร์รองรับโหมดการพิสูจน์ตัวตนแบบ **ใช้ได้เพียงอย่างเดียว** สองโหมด (เรียงตามลำดับความสำคัญ):

- **LiteLLM proxy** — ตั้งค่า `LITELLM_BASE_URL` และ `LITELLM_API_KEY` เพื่อ route ผ่าน LiteLLM proxy (ดูด้านล่าง)
- **โหมด OAuth** — ตั้งค่าเฉพาะ `CLAUDE_CODE_OAUTH_TOKEN` เท่านั้น (ดูด้านล่าง)

อย่ารวมโหมดเข้าด้วยกัน — ใช้เพียงโหมดเดียว

### Claude Max (OAuth)

| ตัวแปร | คำอธิบาย | ตัวอย่าง |
|----------|-------------|---------|
| `CLAUDE_CODE_OAUTH_TOKEN` | OAuth token จากการสมัครสมาชิก Claude Max | `sk-ant-oat01-...` |

เมื่อตั้งค่าแล้ว Claude Code จะพิสูจน์ตัวตนโดยตรงกับ Anthropic ผ่าน OAuth — ไม่ต้องใช้ API key หรือ proxy รับ token ของคุณได้จากการตั้งค่า Claude Code หรือ [คอนโซล Anthropic](https://console.anthropic.com/)

### LiteLLM Proxy

หาก LiteLLM (หรืออื่นๆ) proxy ของคุณรองรับ Anthropic Messages API แบบ native อยู่แล้ว — รวมถึง `web_search`, streaming และ tool use — ให้ตั้งค่า `LITELLM_BASE_URL` เป็นโดเมนของ proxy คอนเทนเนอร์จะคำนวณ URL เฉพาะผู้ให้บริการโดยอัตโนมัติ:

- Anthropic endpoint → `${LITELLM_BASE_URL}/anthropic` (ใช้โดย Claude Code)
- OpenAI-compatible endpoint → `${LITELLM_BASE_URL}/openai/v1`

| ตัวแปร | คำอธิบาย | ตัวอย่าง |
|----------|-------------|---------|
| `LITELLM_BASE_URL` | โดเมนของ LiteLLM proxy ของคุณ (ไม่ต้องใส่ path suffix) | `https://litellm.example.com` |
| `LITELLM_API_KEY` | API key สำหรับการพิสูจน์ตัวตนกับ proxy | `your-api-key` หรือค่าเริ่มต้นเป็น `litellm-proxy` |

ผู้ใช้ต้องตั้งค่าเฉพาะโดเมน — provider path suffix จะถูกเพิ่มโดยอัตโนมัติ
ในเวลารันไทม์ entrypoint จะคำนวณ `ANTHROPIC_API_KEY` จาก `LITELLM_API_KEY` เพื่อให้ Claude Code สามารถใช้ proxy credential ได้โดยไม่ต้องมีตัวแปรแยกต่างหากสำหรับผู้ใช้

ชื่อโมเดล **ไม่ถูก** remap ในโหมดนี้ Claude Code จะส่ง model identifier มาตรฐาน (เช่น `claude-sonnet-4-6`) และ LiteLLM จะ route ไปยัง backend ที่คุณกำหนดค่าไว้

### การตรวจสอบใบรับรอง TLS

โดยค่าเริ่มต้น Node.js จะตรวจสอบใบรับรอง TLS สำหรับการเชื่อมต่อ HTTPS ทั้งหมด หากผู้ให้บริการ API upstream ของคุณใช้ใบรับรองที่ลงนามด้วยตนเอง (พบได้ทั่วไปกับ Open WebUI หรือ LiteLLM proxy ภายใน) คุณสามารถปิดการตรวจสอบได้โดยเพิ่มสิ่งนี้ในไฟล์ `.env`:

```ini
NODE_TLS_REJECT_UNAUTHORIZED=0
```

สิ่งนี้ **ไม่แนะนำ** สำหรับการใช้งานจริง ตั้งค่านี้เฉพาะเมื่อคุณเชื่อถือเส้นทางเครือข่ายระหว่างคอนเทนเนอร์และ API endpoint ของคุณ

### ตัวแทนเขียนโค้ด AI

คอนเทนเนอร์มีตัวแทนเขียนโค้ด AI หลายตัวนอกเหนือจาก Claude Code:

| ตัวแทน | คำสั่ง | ผู้ให้บริการ |

Pi และ Oh-My-Pi ถูกติดตั้งเป็น global npm package Codex เป็น standalone binary ที่อัปเดตตัวเองตอนรันไทม์

คอนเทนเนอร์รัน virtual display stack สำหรับดูและโต้ตอบกับเบราว์เซอร์แบบมีหน้าจอ ดู [จอแสดงผลระยะไกล (noVNC)](./vnc) สำหรับคำแนะนำการเชื่อมต่อและตัวแปรสภาพแวดล้อม (`ENABLE_VNC`, `VNC_RESOLUTION`, `DISPLAY`, `NOVNC_HOST_PORT`)

### Chrome DevTools MCP

คอนเทนเนอร์มี [Chrome DevTools MCP](https://github.com/anthropics/chrome-devtools-mcp) server ที่กำหนดค่าไว้แล้วสำหรับการทำงานอัตโนมัติของเบราว์เซอร์แบบ headless Claude Code สามารถเปิดหน้าเว็บ ถ่ายภาพหน้าจอ และตรวจสอบ DOM ได้โดยไม่ต้องตั้งค่าเพิ่มเติม ดู [Chrome DevTools MCP](./chrome) สำหรับรายละเอียด

### Git

| ตัวแปร | คำอธิบาย | วิธีตั้งค่า |
|----------|-------------|------------|
| `GIT_AUTHOR_NAME` | ชื่อสำหรับ git commit | `git config user.name` |
| `GIT_AUTHOR_EMAIL` | อีเมลสำหรับ git commit | `git config user.email` |

หาก git ถูกกำหนดค่าบนเครื่อง host ของคุณ ให้เติมค่าทั้งสองอัตโนมัติ:

```bash
echo "GIT_AUTHOR_EMAIL=$(git config user.email)" >> .env
echo "GIT_AUTHOR_NAME=\"$(git config user.name)\"" >> .env
```

### GitHub CLI

| ตัวแปร | คำอธิบาย | วิธีตั้งค่า |
|----------|-------------|------------|
| `GH_TOKEN` | GitHub personal access token สำหรับ `gh` CLI และ HTTPS git | `gh auth token` (หากติดตั้ง `gh` ไว้ในเครื่อง) |

เมื่อตั้งค่า `GH_TOKEN` ในไฟล์ `.env` แล้ว `gh` CLI จะพิสูจน์ตัวตนโดยอัตโนมัติ — ไม่ต้องรัน `gh auth login` Entrypoint ยังรัน `gh auth setup-git` ซึ่งกำหนดค่า git credential helper เพื่อให้ HTTPS `git clone` และ `git push` ทำงานได้โดยไม่ต้องใช้ SSH key

หาก `gh` ถูกติดตั้งและพิสูจน์ตัวตนแล้วบนเครื่อง host ของคุณ ให้ดึง token โดยตรง:

```bash
echo "GH_TOKEN=$(gh auth token)" >> .env
```

มิฉะนั้น [สร้าง personal access token](https://github.com/settings/tokens) และเพิ่มด้วยตนเอง ทั้ง fine-grained token (แนะนำ) และ classic token ใช้ได้ Fine-grained token ให้คุณจำกัดขอบเขตการเข้าถึงเฉพาะ repository ที่ต้องการ Classic token ต้องการ scope `repo` สำหรับการเข้าถึง private repository

เพื่อตรวจสอบการพิสูจน์ตัวตนภายในคอนเทนเนอร์:

```bash
gh auth status
```

## การรับรู้เครื่องมือของ Claude Code

คอนเทนเนอร์มีการกำหนดค่าที่ฝังมาแล้วเพื่อป้องกันไม่ให้ Claude Code สับสนชื่อเครื่องมือ การรับรู้เครื่องมือถูกติดตั้งที่ **สอง** ระดับหน่วยความจำเพื่อการป้องกันแบบหลายชั้น:

- **นโยบายที่จัดการ** (`/etc/claude-code/CLAUDE.md`) — ติดตั้งตอน Docker build นี่คือระดับที่มีความสำคัญสูงสุดในลำดับชั้นหน่วยความจำของ Claude Code และจะถูกโหลดเสมอ แม้ว่าจะมี `CLAUDE.md` ระดับโปรเจกต์ขนาดใหญ่อยู่ในไดเรกทอรีที่ทำงาน
- **หน่วยความจำผู้ใช้** (`~/.claude/CLAUDE.md`) — สร้างโดย entrypoint เมื่อเริ่มต้นครั้งแรก ให้เลเยอร์สำรองและสำเนาที่มองเห็นได้และปรับแต่งได้สำหรับผู้ใช้

### ลำดับความสำคัญของระดับหน่วยความจำ

Claude Code โหลดคำสั่งจากหลายระดับหน่วยความจำ (จากสูงสุดไปต่ำสุด):

1. **นโยบายที่จัดการ** (`/etc/claude-code/`) — โหลดเสมอ ไม่สามารถยกเว้นได้
2. **หน่วยความจำโปรเจกต์** (`./CLAUDE.md`) — โหลดตามไดเรกทอรีโปรเจกต์
3. **หน่วยความจำผู้ใช้** (`~/.claude/CLAUDE.md`) — โหลดทั่วโลกสำหรับทุก session
4. **หน่วยความจำในเครื่อง** (`./CLAUDE.local.md`) — การแทนที่ส่วนตัวต่อโปรเจกต์

หากไม่มีระดับนโยบายที่จัดการ การรับรู้เครื่องมือในหน่วยความจำผู้ใช้อาจถูกลดความสำคัญเมื่อแข่งขันกับ `CLAUDE.md` ระดับโปรเจกต์ขนาดใหญ่ ทำให้ Claude สูญเสียการรับรู้ชื่อเครื่องมือแบบ PascalCase ระดับนโยบายที่จัดการรับประกันว่าการรับรู้เครื่องมือจะถูกโหลดที่ลำดับความสำคัญสูงสุดเสมอ

### การทดสอบตัวเอง

รันการทดสอบตัวเองที่มีมาในตัวเพื่อตรวจสอบการกำหนดค่า:

```bash
claude-self-test
```

สิ่งนี้ตรวจสอบว่าไฟล์กำหนดค่าทั้งหมดอยู่ในตำแหน่งที่ถูกต้องและมีการอ้างอิงเครื่องมือที่คาดหวัง

## การเพิ่มเครื่องมือ

หากต้องการเพิ่มเครื่องมือในอิมเมจ ให้แก้ไข `Dockerfile` และ build ใหม่ในเครื่อง ดู [การพัฒนาในเครื่อง — การเพิ่มเครื่องมือ](./local-development#adding-tools) สำหรับคำแนะนำในการเพิ่ม APT package, npm tool, pip tool และการดาวน์โหลด binary

## ส่วนขยาย VS Code

ส่วนขยายถูกกำหนดค่าใน `.devcontainer/devcontainer.json` ภายใต้ `customizations.vscode.extensions` และถูกติดตั้งโดยอัตโนมัติเมื่อเปิดใน VS Code

## การคงอยู่ของข้อมูล

ข้อมูลทั้งหมดอยู่ใน named volume — ไม่มีการ mount ไดเรกทอรี host:

| Volume | จุดเชื่อมต่อ | เนื้อหา |
|--------|-------------|----------|
| `workspace` | `/workspace` | Repository ที่โคลนและไฟล์โปรเจกต์ของคุณ |
| `home` | `/home/vscode` | ประวัติ shell, การกำหนดค่าเครื่องมือ, แคช, SSH key |

## การแมปพอร์ต

| Host | คอนเทนเนอร์ | บริการ |
|------|-----------|---------|
| `127.0.0.1:${NOVNC_HOST_PORT:-6080}` | `6080` | จอแสดงผลระยะไกล noVNC (localhost เท่านั้น, แทนที่ด้วย `NOVNC_HOST_PORT`) |

พอร์ตถูกผูกกับ `127.0.0.1` เพื่อให้เข้าถึงได้จากเครื่องของคุณเท่านั้น ไม่ใช่จากเครือข่าย
