## ความเป็นมา

ในเดือนกุมภาพันธ์ 2025 LayerX Security Research ได้เปิดเผยช่องโหว่การเรียกใช้โค้ดระยะไกลแบบ zero-click ที่ร้ายแรงในเฟรมเวิร์ก Claude Desktop Extensions (DXT) ของ Anthropic ส่วนขยายทำงานโดยไม่มี sandbox ด้วยสิทธิ์ระบบเต็มรูปแบบ ทำหน้าที่เป็นสะพานเชื่อมการทำงานระหว่างโมเดลภาษาและระบบปฏิบัติการในเครื่อง

ทีมรักษาความปลอดภัยขององค์กรจำนวนมากในปัจจุบันกำหนดให้เครื่องมือเขียนโค้ด AI ทำงานเฉพาะภายในเครื่องเสมือน คอนเทนเนอร์ หรือสภาพแวดล้อมที่ใช้แล้วทิ้งเท่านั้น — ไม่ใช่บนอุปกรณ์ปลายทางขององค์กรโดยตรง

**devcontainer นี้ตอบสนองข้อกำหนดดังกล่าว**

## การควบคุมความปลอดภัย

### ไม่มีการเมาท์โวลุ่มของโฮสต์

ข้อมูลทั้งหมดอยู่ใน Docker named volumes — ไม่ใช่ไดเรกทอรีโฮสต์แบบ bind-mount การถูกบุกรุกภายในคอนเทนเนอร์ **ไม่สามารถ** เข้าถึงไฟล์บนเครื่องโฮสต์ของคุณได้ ดูรายละเอียดโวลุ่มที่ [การกำหนดค่า — การจัดเก็บข้อมูลถาวร](./configuration#data-persistence)

### ไม่มีที่เก็บข้อมูลรับรองร่วม

คอนเทนเนอร์ไม่มีการเข้าถึง macOS Keychain, Windows Credential Manager, โปรไฟล์เบราว์เซอร์ หรือ SSH agent ของโฮสต์ คีย์ SSH จะถูกแทรกเข้ามาตอนเริ่มต้นผ่านตัวแปรสภาพแวดล้อมที่เข้ารหัส base64 และมีอยู่เฉพาะภายในคอนเทนเนอร์เท่านั้น

### ไม่มีการซิงค์คลิปบอร์ด

คอนเทนเนอร์ Docker ไม่แชร์สถานะคลิปบอร์ดกับโฮสต์ ข้อมูลไม่สามารถรั่วไหลผ่านการคัดลอก/วาง

### การแยกเครือข่าย

คอนเทนเนอร์สื่อสารกับโลกภายนอกในหนึ่งในสองโหมดการตรวจสอบสิทธิ์ที่รองรับ:

- **โหมด LiteLLM proxy** — HTTPS ขาออกไปยัง LiteLLM proxy ของคุณผ่าน NAT ของ Docker
- **โหมด OAuth** — HTTPS ขาออกโดยตรงไปยัง Anthropic ผ่าน NAT ของ Docker

คอนเทนเนอร์ไม่มีการเข้าถึงอุโมงค์ VPN อินเทอร์เฟซเครือข่ายขององค์กร หรือบริการอื่น ๆ บนโฮสต์

### ออกแบบให้เป็นแบบชั่วคราว

สภาพแวดล้อมทั้งหมดสามารถทำลายและสร้างขึ้นใหม่ได้ในไม่กี่วินาที ให้ถือว่าคอนเทนเนอร์เป็นสิ่งที่ใช้แล้วทิ้ง

### เครื่องมือแบบ CLI เท่านั้น

สภาพแวดล้อมนี้รันเครื่องมือ AI เป็น **อินเทอร์เฟซบรรทัดคำสั่งเท่านั้น** ไม่มีแอปพลิเคชันเดสก์ท็อป ไม่มีส่วนขยายเบราว์เซอร์ ไม่มีเฟรมเวิร์ก DXT ช่องโหว่ที่เป็นสาเหตุของโมเดลการแยกส่วนนี้ไม่สามารถนำไปใช้กับการใช้งาน CLI ภายในคอนเทนเนอร์

### การเสริมความแข็งแกร่งของคอนเทนเนอร์

`docker-compose.yml` ใช้มาตรการป้องกันเชิงลึกหลายประการ:

- **การลดสิทธิ์ (Capability dropping)** — สิทธิ์ Linux ทั้งหมดจะถูกลบออก (`cap_drop: ALL`) จากนั้นเพิ่มกลับเฉพาะชุดขั้นต่ำ: `CHOWN`, `DAC_OVERRIDE`, `FOWNER` (สำหรับแก้ไขสิทธิ์โวลุ่มใน entrypoint), `SETUID`/`SETGID` (สำหรับ sudo), `NET_RAW` (สำหรับ ping/tcpdump) และ `NET_ADMIN` (สำหรับเครื่องมือจับแพ็กเก็ตเช่น Wireshark และ bettercap)
- **การจำกัดทรัพยากร** — CPU (4 คอร์), หน่วยความจำ (8 GB) และจำนวน PID (4096) ถูกจำกัดผ่าน `mem_limit`, `cpus` และ `pids_limit` เพื่อป้องกันการโจมตีแบบทำให้ทรัพยากรหมด
- **tmpfs สำหรับ `/tmp`** — ไดเรกทอรี `/tmp` ถูกเมาท์เป็น tmpfs ขนาด 256 MB ทำให้ไฟล์ชั่วคราวอยู่ในหน่วยความจำและป้องกันไม่ให้คงอยู่ในเลเยอร์ที่เขียนได้ของคอนเทนเนอร์
- **การตรวจสอบ TLS เปิดใช้งานโดยค่าเริ่มต้น** — `NODE_TLS_REJECT_UNAUTHORIZED` ไม่ได้ถูกตั้งค่า ดังนั้น Node.js จะตรวจสอบใบรับรอง TLS โดยค่าเริ่มต้น ผู้ใช้ที่ต้องการยอมรับใบรับรองที่ลงนามเอง (เช่น สำหรับ proxy ภายใน) สามารถเลือกเปิดใช้งานโดยเพิ่ม `NODE_TLS_REJECT_UNAUTHORIZED=0` ในไฟล์ `.env` ของตน
- **umask ของคีย์ SSH** — คีย์ส่วนตัว SSH ถูกเขียนภายใต้ `umask 077` ดังนั้นไฟล์จะไม่สามารถอ่านได้โดยทุกคน — แม้แต่ช่วงเวลาสั้น ๆ — ระหว่างการสร้างและการเรียก `chmod`

### เครื่องมือทดสอบความปลอดภัยและการเจาะระบบ

คอนเทนเนอร์มีเครื่องมือทดสอบความปลอดภัยและการเจาะระบบที่ติดตั้งล่วงหน้าประมาณ 80 รายการสำหรับสถานการณ์การทดสอบที่ได้รับอนุญาต เครื่องมือจัดเรียงตามหมวดหมู่:

| หมวดหมู่ | เครื่องมือ (ตัวอย่าง) |
|----------|-----------------|
| การวิเคราะห์เครือข่าย | tshark, wireshark, masscan, hping3, bettercap (amd64), netdiscover |
| เครื่องสแกนเว็บ | nikto, sqlmap, dirb, whatweb, sslscan, OWASP ZAP, dalfox, feroxbuster |
| รหัสผ่านและการตรวจสอบสิทธิ์ | hydra, john, hashcat, medusa, ncrack |
| วิศวกรรมย้อนกลับ | radare2, Ghidra, gdb, binwalk, strace, ltrace |
| การสำรวจ | subfinder, amass, httpx, nuclei, gau, waybackurls, recon-ng, spiderfoot |
| การฟัซซิ่งและการแจกแจง | ffuf, gobuster, SecLists |
| ซัพพลายเชนและความลับ | trufflehog, grype, syft, gitleaks, trivy |
| เฟรมเวิร์กการโจมตี | Metasploit (amd64), searchsploit (ExploitDB) |

เครื่องมือบางตัว (bettercap, Metasploit) มีเฉพาะบน amd64 เนื่องจากข้อจำกัดของแพ็กเกจต้นน้ำ เครื่องมือจับแพ็กเก็ต (tshark, bettercap) ต้องการสิทธิ์ `NET_ADMIN` ซึ่งรวมอยู่ใน `docker-compose.yml` ค่าเริ่มต้น

สำหรับรายการทั้งหมด ให้รัน `claude-self-test` ภายในคอนเทนเนอร์ (ส่วนที่ 7 ตรวจสอบเครื่องมือความปลอดภัยทั้งหมด)

## รายการตรวจสอบการปฏิบัติตามข้อกำหนด

| ข้อกำหนด | สถานะ |
|---|---|
| ไม่ได้ติดตั้งเครื่องมือ AI บนอุปกรณ์ปลายทางโฮสต์ | ใช่ |
| ทำงานในเครื่องเสมือนหรือคอนเทนเนอร์ที่แยกส่วน | ใช่ |
| ไม่มีการเข้าถึงไฟล์แชร์ของโฮสต์ | ใช่ |
| ไม่มีการซิงค์คลิปบอร์ด | ใช่ |
| ไม่มีที่เก็บข้อมูลรับรองร่วม | ใช่ |
| ไม่มี VPN split tunneling เข้าเครือข่ายโปรดักชัน | ใช่ |
| สภาพแวดล้อมเป็นแบบใช้แล้วทิ้ง | ใช่ |
| ไม่มีส่วนขยายหรือปลั๊กอินเดสก์ท็อป | ใช่ |
| ลดสิทธิ์ Linux (สิทธิ์ขั้นต่ำ) | ใช่ |
| บังคับใช้การจำกัดทรัพยากร (CPU/หน่วยความจำ/PIDs) | ใช่ |
| เปิดใช้งานการตรวจสอบใบรับรอง TLS โดยค่าเริ่มต้น | ใช่ |
| บริการภายในผูกกับ localhost เท่านั้น | ใช่ |

## แนวปฏิบัติที่ดีที่สุด

1. **อย่าติดตั้งเครื่องมือเขียนโค้ด AI บนเครื่องโฮสต์ของคุณ** ใช้คอนเทนเนอร์นี้แทน
2. **อย่าเพิ่ม bind mounts** ใน `docker-compose.yml` การกำหนดค่า named-volume เป็นไปโดยตั้งใจ
3. **หมุนเวียนคีย์ SSH** ที่ใช้ภายในคอนเทนเนอร์เป็นระยะ
4. **ตรวจสอบ `.env` ก่อนแชร์** มันมีคีย์ API และอาจมีคีย์ส่วนตัว SSH
5. **อัปเดตเป็นระยะ** เพื่อรับแพตช์ความปลอดภัย:
