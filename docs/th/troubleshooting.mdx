### Claude แสดง "Not logged in"

entrypoint จะตั้งค่า `~/.claude.json` พร้อมแฟล็กการเริ่มต้นใช้งาน หากไฟล์หายไป:

```bash
echo '{"hasCompletedOnboarding": true}' > ~/.claude.json
```

### Claude แสดง "Invalid API key"

ตรวจสอบค่าที่ตั้งไว้จริง:

```bash
echo "$ANTHROPIC_API_KEY"
```

สาเหตุที่พบบ่อย:

- **คีย์ LiteLLM proxy หายไปหรือไม่ถูกต้อง** — หากคุณใช้โหมด proxy ตรวจสอบให้แน่ใจว่า `LITELLM_API_KEY` ใน `.env` มีข้อมูลรับรอง proxy ที่ถูกต้อง จากนั้นรีสตาร์ทคอนเทนเนอร์เพื่อให้ entrypoint สามารถสร้าง `ANTHROPIC_API_KEY` ได้
- **OAuth token หรือโหมด proxy ไม่ตรงกัน** — ใช้โหมดการยืนยันตัวตนเพียงโหมดเดียว หากตั้งค่า `CLAUDE_CODE_OAUTH_TOKEN` ไว้ จะมีความสำคัญเหนือกว่าโหมด LiteLLM proxy

### คำขอ API ล้มเหลวด้วยข้อผิดพลาด 401

ตรวจสอบว่า API key ของคุณตั้งค่าถูกต้อง:

```bash
echo "$ANTHROPIC_API_KEY"
```

ยืนยันว่าใช้งานได้โดยเรียก API โดยตรงไปยังผู้ให้บริการของคุณ

### Claude แจ้ง "model not found"

โมเดลถูกกำหนดโดยผู้ให้บริการและ API key ของคุณ ตรวจสอบเอกสารของผู้ให้บริการสำหรับโมเดลที่ใช้ได้

## GitHub CLI

### "gh: not authenticated"

`gh` CLI ต้องการตัวแปรสภาพแวดล้อม `GH_TOKEN` เพิ่มลงในไฟล์ `.env` ของคุณ:

```ini
GH_TOKEN=ghp_your-token-here
```

สร้างโทเค็นได้ที่ [github.com/settings/tokens](https://github.com/settings/tokens) ทั้ง fine-grained tokens (แนะนำ) หรือ classic tokens ที่มีสิทธิ์ `repo` สามารถใช้ได้ รีสตาร์ทคอนเทนเนอร์หลังจากอัปเดต `.env`

### HTTPS git clone ล้มเหลวด้วยข้อผิดพลาด 401

เมื่อตั้งค่า `GH_TOKEN` แล้ว entrypoint จะรัน `gh auth setup-git` เพื่อกำหนดค่า git credential helper สำหรับ HTTPS หาก HTTPS ทำงานไม่ได้:

1. ตรวจสอบว่าโทเค็นยังใช้ได้: `gh auth status`
2. ตรวจสอบว่า credential helper ถูกกำหนดค่าแล้ว: `git config --global credential.helper`
3. รีสตาร์ทคอนเทนเนอร์เพื่อรัน entrypoint อีกครั้ง

### SSH เทียบกับ HTTPS

วิธีการยืนยันตัวตนทั้งสองแบบสามารถใช้งานร่วมกันในคอนเทนเนอร์ได้:

- **SSH** (`SSH_PRIVATE_KEY` ใน `.env`) — ใช้ URL แบบ `git@github.com:` จำเป็นหากองค์กรของคุณบังคับใช้ SSH
- **HTTPS** (`GH_TOKEN` ใน `.env`) — ใช้ URL แบบ `https://github.com/` ตั้งค่าง่ายกว่า ไม่ต้องจัดการคีย์

หากกำหนดค่าทั้งสองไว้ git จะใช้โปรโตคอลที่ตรงกับ remote URL หากต้องการเปลี่ยน clone ที่มีอยู่:

```bash
# SSH to HTTPS
git remote set-url origin https://github.com/owner/repo.git

# HTTPS to SSH
git remote set-url origin git@github.com:owner/repo.git
```

## ปัญหาเรื่องสิทธิ์

### "Permission denied" บน /workspace หรือ /home

```bash
sudo chown -R $(id -u):$(id -g) /workspace ~
```

### npm install ล้มเหลวด้วยข้อผิดพลาด EACCES

entrypoint จะกำหนดค่า npm global prefix ที่ผู้ใช้สามารถเขียนได้โดยอัตโนมัติที่ `~/.npm-global` ดังนั้นคำสั่ง `npm install -g` ขณะรันไทม์ควรทำงานได้โดยไม่ต้องใช้ `sudo` หากยังพบข้อผิดพลาด EACCES ให้แก้ไขด้วยตนเอง:

```bash
mkdir -p ~/.npm-global
npm config set prefix ~/.npm-global
export PATH="$HOME/.npm-global/bin:$PATH"
```

## เครือข่าย

### ไม่สามารถเข้าถึงอินเทอร์เน็ตจากภายในคอนเทนเนอร์

```bash
ping -c 1 8.8.8.8
nslookup google.com
curl -I https://github.com
```

<Tabs syncKey="runtime">
<TabItem label="Docker">

หาก DNS ล้มเหลว ให้เพิ่มในไฟล์ `/etc/docker/daemon.json` บนเครื่องโฮสต์:

```json
{
  "dns": ["8.8.8.8", "8.8.4.4"]
}
```

จากนั้นรีสตาร์ท Docker

</TabItem>
<TabItem label="Podman">

หาก DNS ล้มเหลว ให้กำหนดค่า DNS ภายใน Podman machine:

```bash
podman machine ssh
sudo sh -c 'echo "nameserver 8.8.8.8" > /etc/resolv.conf'
exit
```

หรือเพิ่มการตั้งค่า DNS ในไฟล์ `~/.config/containers/containers.conf` บนเครื่องโฮสต์:

```ini
[containers]
dns_servers = ["8.8.8.8", "8.8.4.4"]
```

จากนั้นรีสตาร์ท Podman: `podman machine stop && podman machine start`

</TabItem>

## การแสดงผลระยะไกล (noVNC)

### หน้าจอดำใน noVNC

Xvfb อาจยังไม่ได้เริ่มทำงาน ตรวจสอบภายในคอนเทนเนอร์:

```bash
ps aux | grep Xvfb
```

หากไม่ได้ทำงาน ตรวจสอบว่า `ENABLE_VNC` ไม่ได้ตั้งค่าเป็น `false` และรีสตาร์ทคอนเทนเนอร์

### ข้อผิดพลาด "Cannot open display" หรือ "No display"

ตัวแปรสภาพแวดล้อม `DISPLAY` อาจไม่ได้ตั้งค่า ตรวจสอบ:

```bash
echo "$DISPLAY"
```

ควรจะเป็น `:99` หากว่างเปล่า ให้เพิ่ม `DISPLAY=:99` ในไฟล์ `.env` หรือ `devcontainer.json` แล้วรีสตาร์ท

### พอร์ต 6080 ถูกปฏิเสธการเชื่อมต่อ

noVNC อาจไม่ได้ทำงาน ตรวจสอบภายในคอนเทนเนอร์:

```bash
ps aux | grep -E 'novnc|websockify'
```

หากไม่ได้ทำงาน ตรวจสอบว่า `ENABLE_VNC` เป็น `true` (ค่าเริ่มต้น) และตรวจสอบล็อกของคอนเทนเนอร์:

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash
docker compose logs dev | grep -i vnc
```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose logs dev | grep -i vnc
```

</TabItem>
</Tabs>

### พอร์ต 6080 ถูกใช้งานอยู่แล้ว

เปลี่ยนพอร์ตโฮสต์ใน `docker-compose.yml`:

```yaml
ports:
  - "127.0.0.1:16080:6080"
```

## Chrome DevTools MCP

### Chrome remote debugging ไม่ตอบสนอง

อินสแตนซ์ Chrome ที่ใช้ร่วมกันควรทำงานบนพอร์ต 9222 ตรวจสอบภายในคอนเทนเนอร์:

```bash
# Is Chrome running?
curl http://localhost:9222/json/version

# Check Chrome logs
cat ~/.local/share/chrome-browser/chrome.log

# Verify the symlink
ls -la /opt/google/chrome/chrome

# Restart Chrome manually
. /usr/local/lib/chrome-browser.sh
start_chrome_browser
```

หาก symlink หายไป ให้ดึงอิมเมจล่าสุดและรีสตาร์ทคอนเทนเนอร์

### ข้อผิดพลาด browser profile lock

ด้วยสถาปัตยกรรมเบราว์เซอร์ที่ใช้ร่วมกัน ข้อผิดพลาด profile lock ไม่ควรเกิดขึ้น หากพบปัญหานี้ อาจมีกระบวนการ Chrome ที่ค้างอยู่ถือ lock ไว้:

```bash
# Kill any stale Chrome processes
pkill -f 'chrome.*remote-debugging-port' || true

# Remove the lock file
rm -f ~/.cache/chrome-devtools-mcp/chrome-profile/SingletonLock

# Restart Chrome
. /usr/local/lib/chrome-browser.sh
start_chrome_browser
```

### เครื่องมือ chrome-devtools-mcp ซ้ำกัน

เซิร์ฟเวอร์ chrome-devtools-mcp ถูกลงทะเบียนในระดับ global ใน `~/.claude/settings.json` หากคุณมี `.mcp.json` ระดับ repo ที่ลงทะเบียนไว้ด้วย คุณจะเห็นเครื่องมือซ้ำกัน ลบรายการ `chrome-devtools-mcp` ออกจากไฟล์ `.mcp.json` ระดับ repo:

```bash
# Check for per-repo config
cat .mcp.json
```

ลบคีย์ `chrome-devtools-mcp` ออกจาก `.mcp.json` ระดับ repo เพื่อใช้การลงทะเบียนแบบ global
