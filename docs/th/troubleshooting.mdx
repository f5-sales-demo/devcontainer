### Claude แสดง "Not logged in"

Entrypoint จะตั้งค่า `~/.claude.json` พร้อมแฟล็กการเริ่มต้นใช้งาน หากไฟล์หายไป:

```bash
echo '{"hasCompletedOnboarding": true}' > ~/.claude.json
```

### Claude แสดง "Invalid API key"

ตรวจสอบค่าที่ตั้งอยู่จริง:

```bash
echo "$ANTHROPIC_API_KEY"
```

สาเหตุที่พบบ่อย:

- **คีย์ LiteLLM proxy หายไปหรือไม่ถูกต้อง** — หากคุณใช้โหมด proxy ตรวจสอบว่า `LITELLM_API_KEY` ในไฟล์ `.env` มีข้อมูลรับรอง proxy ที่ถูกต้อง จากนั้นรีสตาร์ทคอนเทนเนอร์เพื่อให้ entrypoint สามารถสร้าง `ANTHROPIC_API_KEY` ได้
- **โทเค็น OAuth หรือโหมด proxy ไม่ตรงกัน** — ใช้โหมดการยืนยันตัวตนเพียงโหมดเดียว หาก `CLAUDE_CODE_OAUTH_TOKEN` ถูกตั้งค่าไว้ จะมีลำดับความสำคัญเหนือโหมด LiteLLM proxy

### คำขอ API ล้มเหลวพร้อมข้อผิดพลาด 401

ตรวจสอบว่า API key ของคุณถูกตั้งค่าอย่างถูกต้อง:

```bash
echo "$ANTHROPIC_API_KEY"
```

ตรวจสอบให้แน่ใจว่าใช้งานได้โดยเรียกใช้งานโดยตรงไปยังผู้ให้บริการของคุณ

### Claude แจ้งว่า "model not found"

โมเดลถูกกำหนดโดยผู้ให้บริการและ API key ของคุณ ตรวจสอบเอกสารของผู้ให้บริการสำหรับโมเดลที่ใช้งานได้

## GitHub CLI

### "gh: not authenticated"

`gh` CLI ต้องการตัวแปรสภาพแวดล้อม `GH_TOKEN` เพิ่มลงในไฟล์ `.env` ของคุณ:

```ini
GH_TOKEN=ghp_your-token-here
```

สร้างโทเค็นได้ที่ [github.com/settings/tokens](https://github.com/settings/tokens) โทเค็นแบบ Fine-grained (แนะนำ) หรือโทเค็นแบบ Classic ที่มีสิทธิ์ `repo` สามารถใช้งานได้ทั้งคู่ รีสตาร์ทคอนเทนเนอร์หลังจากอัปเดต `.env`

### HTTPS git clone ล้มเหลวพร้อมข้อผิดพลาด 401

เมื่อ `GH_TOKEN` ถูกตั้งค่า entrypoint จะรันคำสั่ง `gh auth setup-git` เพื่อกำหนดค่า git credential helper สำหรับ HTTPS หากการดำเนินการ HTTPS ล้มเหลว:

1. ตรวจสอบว่าโทเค็นยังใช้งานได้: `gh auth status`
2. ตรวจสอบว่า credential helper ถูกกำหนดค่าแล้ว: `git config --global credential.helper`
3. รีสตาร์ทคอนเทนเนอร์เพื่อรัน entrypoint ใหม่

### SSH เทียบกับ HTTPS

วิธีการยืนยันตัวตนทั้งสองแบบสามารถใช้ร่วมกันในคอนเทนเนอร์ได้:

- **SSH** (`SSH_PRIVATE_KEY` ในไฟล์ `.env`) — ใช้ URL แบบ `git@github.com:` จำเป็นหากองค์กรของคุณบังคับใช้ SSH
- **HTTPS** (`GH_TOKEN` ในไฟล์ `.env`) — ใช้ URL แบบ `https://github.com/` ตั้งค่าง่ายกว่า ไม่ต้องจัดการคีย์

หากกำหนดค่าทั้งสองแบบ git จะใช้โปรโตคอลที่ตรงกับ remote URL เพื่อสลับ clone ที่มีอยู่:

```bash
# SSH เป็น HTTPS
git remote set-url origin https://github.com/owner/repo.git

# HTTPS เป็น SSH
git remote set-url origin git@github.com:owner/repo.git
```

## ปัญหาสิทธิ์การเข้าถึง

### "Permission denied" บน /workspace หรือ /home

```bash
sudo chown -R $(id -u):$(id -g) /workspace ~
```

### npm install ล้มเหลวพร้อมข้อผิดพลาด EACCES

Entrypoint จะกำหนดค่า npm global prefix ที่ผู้ใช้สามารถเขียนได้โดยอัตโนมัติที่ `~/.npm-global` ดังนั้นคำสั่ง `npm install -g` ขณะรันไทม์ควรทำงานได้โดยไม่ต้องใช้ `sudo` หากคุณยังเห็นข้อผิดพลาด EACCES ให้แก้ไขด้วยตนเอง:

```bash
mkdir -p ~/.npm-global
npm config set prefix ~/.npm-global
export PATH="$HOME/.npm-global/bin:$PATH"
```

## เครือข่าย

### ไม่สามารถเข้าถึงอินเทอร์เน็ตจากภายในคอนเทนเนอร์

```bash
ping -c 1 8.8.8.8
nslookup google.com
curl -I https://github.com
```

<Tabs syncKey="runtime">
<TabItem label="Docker">

หาก DNS ล้มเหลว ให้เพิ่มลงใน `/etc/docker/daemon.json` บนเครื่องโฮสต์:

```json
{
  "dns": ["8.8.8.8", "8.8.4.4"]
}
```

จากนั้นรีสตาร์ท Docker

</TabItem>
<TabItem label="Podman">

หาก DNS ล้มเหลว ให้กำหนดค่า DNS ภายใน Podman machine:

```bash
podman machine ssh
sudo sh -c 'echo "nameserver 8.8.8.8" > /etc/resolv.conf'
exit
```

หรือเพิ่มการตั้งค่า DNS ลงใน `~/.config/containers/containers.conf` บนเครื่องโฮสต์:

```ini
[containers]
dns_servers = ["8.8.8.8", "8.8.4.4"]
```

จากนั้นรีสตาร์ท Podman: `podman machine stop && podman machine start`

</TabItem>
</Tabs>

## จอแสดงผลระยะไกล (noVNC)

### หน้าจอดำใน noVNC

Xvfb อาจไม่ได้เริ่มทำงาน ตรวจสอบภายในคอนเทนเนอร์:

```bash
ps aux | grep Xvfb
```

หากไม่ทำงาน ตรวจสอบว่า `ENABLE_VNC` ไม่ได้ถูกตั้งค่าเป็น `false` และรีสตาร์ทคอนเทนเนอร์

### ข้อผิดพลาด "Cannot open display" หรือ "No display"

ตัวแปรสภาพแวดล้อม `DISPLAY` อาจไม่ได้ถูกตั้งค่า ตรวจสอบ:

```bash
echo "$DISPLAY"
```

ค่าควรเป็น `:99` หากว่างเปล่า ให้เพิ่ม `DISPLAY=:99` ลงในไฟล์ `.env` หรือ `devcontainer.json` และรีสตาร์ท

### พอร์ต 6080 ปฏิเสธการเชื่อมต่อ

noVNC อาจไม่ได้ทำงานอยู่ ตรวจสอบภายในคอนเทนเนอร์:

```bash
ps aux | grep -E 'novnc|websockify'
```

หากไม่ทำงาน ตรวจสอบว่า `ENABLE_VNC` เป็น `true` (ค่าเริ่มต้น) และตรวจสอบบันทึกคอนเทนเนอร์:

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash
docker compose logs dev | grep -i vnc
```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose logs dev | grep -i vnc
```

</TabItem>
</Tabs>

### พอร์ต 6080 ถูกใช้งานอยู่แล้ว

เปลี่ยนพอร์ตโฮสต์ในไฟล์ `docker-compose.yml`:

```yaml
ports:
  - "127.0.0.1:16080:6080"
```

## Chrome DevTools MCP

### Chrome remote debugging ไม่ตอบสนอง

อินสแตนซ์ Chrome ที่ใช้ร่วมกันควรทำงานอยู่บนพอร์ต 9222 ตรวจสอบภายในคอนเทนเนอร์:

```bash
# Chrome ทำงานอยู่หรือไม่?
curl http://localhost:9222/json/version

# ตรวจสอบบันทึก Chrome
cat ~/.local/share/chrome-browser/chrome.log

# ตรวจสอบ symlink
ls -la /opt/google/chrome/chrome

# รีสตาร์ท Chrome ด้วยตนเอง
. /usr/local/lib/chrome-browser.sh
start_chrome_browser
```

หาก symlink หายไป ให้ดึงอิมเมจล่าสุดและรีสตาร์ทคอนเทนเนอร์

### ข้อผิดพลาด Browser profile lock

ด้วยสถาปัตยกรรมเบราว์เซอร์แบบใช้ร่วมกัน ข้อผิดพลาด profile lock ไม่ควรเกิดขึ้น หากคุณพบข้อผิดพลาดนี้ อาจมีกระบวนการ Chrome ที่ค้างอยู่กำลังล็อกไฟล์:

```bash
# ปิดกระบวนการ Chrome ที่ค้างอยู่
pkill -f 'chrome.*remote-debugging-port' || true

# ลบไฟล์ lock
rm -f ~/.cache/chrome-devtools-mcp/chrome-profile/SingletonLock

# รีสตาร์ท Chrome
. /usr/local/lib/chrome-browser.sh
start_chrome_browser
```

### เครื่องมือ chrome-devtools-mcp ซ้ำกัน

เซิร์ฟเวอร์ chrome-devtools-mcp ถูกลงทะเบียนแบบ global ใน `~/.claude/settings.json` หากคุณมีไฟล์ `.mcp.json` ระดับ repo ที่ลงทะเบียนเช่นกัน คุณจะเห็นเครื่องมือซ้ำกัน ให้ลบรายการ `chrome-devtools-mcp` ออกจากไฟล์ `.mcp.json` ระดับ repo:

```bash
# ตรวจสอบการกำหนดค่าระดับ repo
cat .mcp.json
```

ลบคีย์ `chrome-devtools-mcp` ออกจากไฟล์ `.mcp.json` ระดับ repo เพื่อใช้การลงทะเบียนแบบ global
