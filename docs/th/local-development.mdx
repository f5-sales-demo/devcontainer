หน้านี้สำหรับนักพัฒนาที่ต้องการโคลนรีโพซิทอรี บิลด์ image ในเครื่อง หรือปรับแต่ง Dockerfile หากคุณต้องการใช้คอนเทนเนอร์ที่บิลด์ไว้แล้วเท่านั้น ดูที่ [เริ่มต้นใช้งาน](./getting-started)

## โคลนรีโพซิทอรี

```bash
git clone https://github.com/f5xc-salesdemos/devcontainer.git
cd devcontainer
```

## โครงสร้างโปรเจกต์

```text
.
├── Dockerfile                  # Multi-stage image build
├── docker-compose.yml          # Base compose — pulls pre-built image
├── docker-compose.build.yml    # Build override — use explicitly with -f
├── .devcontainer/              # VS Code Dev Container config
│   └── devcontainer.json
├── entrypoint.sh               # Container startup script
├── .env.example                # Example environment variables
├── docs/                       # Documentation site (Starlight)
├── claude-config/              # Claude Code configuration files
├── codex-config/               # Codex configuration
├── pi-config/                  # Pi configuration
└── omp-config/                 # Oh-My-Pi (omp) configuration
```

## ไฟล์บิลด์

ไฟล์ `docker-compose.build.yml` เพิ่มการรองรับการบิลด์ในเครื่อง ไฟล์นี้**ไม่**ถูกรวมอัตโนมัติ — คุณต้องระบุอย่างชัดเจนด้วยแฟล็ก `-f` ซึ่งหมายความว่า `docker compose up -d` (หรือ `podman-compose up -d`) จะดึง image ที่บิลด์ไว้แล้วจาก GHCR เสมอ ไม่ว่าคุณจะโคลนรีโพหรือไม่ก็ตาม

<Tabs syncKey="runtime">
<TabItem label="Docker">

คุณสามารถตรวจสอบว่าคำสั่ง `docker compose up` ธรรมดาใช้เฉพาะ image ที่บิลด์ไว้แล้วเท่านั้น (ไม่มี `build:` context):

```bash
docker compose config
```

เปรียบเทียบกับการกำหนดค่าบิลด์แบบระบุชัดเจน:

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml config
```

</TabItem>
<TabItem label="Podman">

คุณสามารถตรวจสอบว่าคำสั่ง `podman-compose up` ธรรมดาใช้เฉพาะ image ที่บิลด์ไว้แล้วเท่านั้น (ไม่มี `build:` context):

```bash
podman-compose config
```

เปรียบเทียบกับการกำหนดค่าบิลด์แบบระบุชัดเจน:

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml config
```

</TabItem>
</Tabs>


</TabItem>

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

คำสั่งนี้จะรวมไฟล์บิลด์เข้ากับไฟล์ compose หลัก บิลด์ image จาก `Dockerfile` ในเครื่อง และเริ่มต้นคอนเทนเนอร์ แฟล็ก `--build` บังคับให้บิลด์ใหม่แม้ว่าจะมี image ที่แคชไว้อยู่แล้ว

## สถาปัตยกรรมบิลด์แบบสองขั้นตอน

Dockerfile ใช้การบิลด์แบบสองขั้นตอนที่ปรับให้เหมาะสมกับ Docker layer caching:

### ขั้นตอนที่ 1: `deps` (รากฐานที่เสถียร, ~4.5 GB)

เลเยอร์เหล่านี้จะบิลด์ใหม่เฉพาะเมื่อมีการเปลี่ยนเวอร์ชัน ARG หรือแพ็กเกจ APT เปลี่ยนแปลงเท่านั้น เมื่อมีการเปลี่ยนแปลงเครื่องมือทั่วไป BuildKit จะข้ามขั้นตอนนี้ทั้งหมด

| ส่วน | เนื้อหา |
|---------|----------|
| 1. APT repos | NodeSource, deadsnakes, HashiCorp, GitHub, Docker, Microsoft, Google Cloud, Dart SDK |

## การเพิ่มเครื่องมือ

แก้ไข `Dockerfile` และเพิ่มเครื่องมือของคุณในส่วนที่เหมาะสม บิลด์ใหม่หลังจากเพิ่มเสร็จ:

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
</Tabs>

หรือใน VS Code: **Dev Containers &rarr; Rebuild Container**

### แพ็กเกจ APT

เพิ่มในบล็อก `apt-get install` ส่วนที่ 2:

```dockerfile
RUN apt-get update && apt-get install -y --no-install-recommends \
    your-package-here \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*
```

### เครื่องมือ npm

เพิ่มในบล็อก `npm install -g` ส่วนที่ 11:

```dockerfile
RUN npm install -g \
    your-npm-tool
```

### เครื่องมือ pip

เพิ่มในบล็อก `pip install` ส่วนที่ 12:

```dockerfile
RUN pip install --no-cache-dir --break-system-packages \
    your-python-tool
```

### การดาวน์โหลดไบนารี

เพิ่มในส่วนที่ 10 หรือสร้างเลเยอร์ `RUN` ใหม่พร้อมการตรวจจับสถาปัตยกรรม:

```dockerfile
RUN ARCH=$(dpkg --print-architecture) \
    && curl -fsSL "https://example.com/tool-linux-${ARCH}.tar.gz" \
       | tar -xz -C /usr/local/bin tool
```

## แคชการบิลด์

CI ใช้ **registry-based caching** — เลเยอร์การบิลด์ถูกเก็บใน GHCR (`ghcr.io/f5xc-salesdemos/devcontainer:cache-*`) แทนที่จะเป็นแคชของ GitHub Actions ซึ่งช่วยหลีกเลี่ยงขีดจำกัดแคช GHA 10 GB ที่ทำให้เกิดการบิลด์ใหม่ทั้งหมดบ่อยครั้งสำหรับ image แบบ multi-architecture ขนาด ~6 GB นี้

สถาปัตยกรรมแบบสองขั้นตอนทำงานร่วมกับ registry cache ดังนี้:

- **การอัปเดตเวอร์ชันเครื่องมือ** (npm, pip, binary tools) — บิลด์ใหม่เฉพาะขั้นตอน `final` (~5 นาที)
- **การเปลี่ยนแปลงไฟล์กำหนดค่า** (`claude-config/`, `entrypoint.sh`) — บิลด์ใหม่เฉพาะเลเยอร์สุดท้ายไม่กี่เลเยอร์ (~1 นาที)
- **การเปลี่ยนแปลงรากฐาน** (แพ็กเกจ APT, เวอร์ชัน Go/Rust/Java) — บิลด์ใหม่ทั้งหมดทั้งสองขั้นตอน (~30 นาที)
