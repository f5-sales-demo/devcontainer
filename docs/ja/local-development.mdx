このページは、リポジトリをクローンし、イメージをローカルでビルドしたり、Dockerfileをカスタマイズしたい開発者向けです。ビルド済みコンテナを使用するだけであれば、[はじめに](./getting-started)を参照してください。

## リポジトリのクローン

```bash
git clone https://github.com/f5xc-salesdemos/devcontainer.git
cd devcontainer
```

## プロジェクト構成

```text
.
├── Dockerfile                  # Multi-stage image build
├── docker-compose.yml          # Base compose — pulls pre-built image
├── docker-compose.build.yml    # Build override — use explicitly with -f
├── .devcontainer/              # VS Code Dev Container config
│   └── devcontainer.json
├── entrypoint.sh               # Container startup script
├── .env.example                # Example environment variables
├── docs/                       # Documentation site (Starlight)
├── claude-config/              # Claude Code configuration files
├── codex-config/               # Codex configuration
├── pi-config/                  # Pi configuration
└── omp-config/                 # Oh-My-Pi (omp) configuration
```

## ビルドファイル

`docker-compose.build.yml` ファイルはローカルビルドのサポートを追加します。これは自動マージ**されません** — `-f` フラグで明示的に指定する必要があります。つまり、`docker compose up -d`（または `podman-compose up -d`）は、リポジトリをクローンしたかどうかに関係なく、常にGHCRからビルド済みイメージをプルします。

<Tabs syncKey="runtime">
<TabItem label="Docker">

通常の `docker compose up` がビルド済みイメージのみを使用する（`build:` コンテキストがない）ことを確認できます：

```bash
docker compose config
```

明示的なビルド構成と比較します：

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml config
```

</TabItem>

```bash

podman-compose -f docker-compose.yml -f docker-compose.build.yml config
```

## ローカルでのビルド

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
</Tabs>

これにより、ビルドファイルがベースのcomposeファイルとマージされ、ローカルの `Dockerfile` からイメージがビルドされ、コンテナが起動します。`--build` フラグは、キャッシュされたイメージが存在する場合でも強制的にリビルドします。

## 2ステージビルドアーキテクチャ

DockerfileはDockerレイヤーキャッシュに最適化された2ステージビルドを使用しています：

これらのレイヤーは、バージョンARGが更新されるか、APTパッケージが変更された場合にのみリビルドされます。通常のツールのみの変更では、BuildKitはこのステージ全体をスキップします。

## ツールの追加

`Dockerfile` を編集し、適切なセクションにツールを追加します。追加後にリビルドしてください：

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
</Tabs>

または VS Code で: **Dev Containers &rarr; Rebuild Container**。

### APTパッケージ

セクション2の `apt-get install` ブロックに追加します：

```dockerfile
RUN apt-get update && apt-get install -y --no-install-recommends \
    your-package-here \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*
```

### npmツール

セクション11の `npm install -g` ブロックに追加します：

```dockerfile
RUN npm install -g \
    your-npm-tool
```

### pipツール

セクション12の `pip install` ブロックに追加します：

```dockerfile
RUN pip install --no-cache-dir --break-system-packages \
    your-python-tool
```

### バイナリダウンロード

セクション10に追加するか、アーキテクチャ検出を含む新しい `RUN` レイヤーを作成します：

```dockerfile
RUN ARCH=$(dpkg --print-architecture) \
    && curl -fsSL "https://example.com/tool-linux-${ARCH}.tar.gz" \
       | tar -xz -C /usr/local/bin tool
```

## ビルドキャッシュ

CIは**レジストリベースのキャッシュ**を使用しています — ビルドレイヤーはGitHub Actionsキャッシュではなく、GHCR（`ghcr.io/f5xc-salesdemos/devcontainer:cache-*`）に保存されます。これにより、この約6 GBのマルチアーキテクチャイメージで頻繁に完全リビルドを引き起こしていた10 GBのGHAキャッシュ制限を回避できます。

2ステージアーキテクチャはレジストリキャッシュと連携して、以下のように動作します：

- **ツールバージョンの更新**（npm、pip、バイナリツール）— `final` ステージのみリビルド（約5分）
- **設定ファイルの変更**（`claude-config/`、`entrypoint.sh`）— 最後の数レイヤーのみリビルド（約1分）
- **基盤の変更**（APTパッケージ、Go/Rust/Javaバージョン）— 両ステージの完全リビルド（約30分）
