Esta página é destinada a desenvolvedores que desejam clonar o repositório, compilar a imagem localmente ou personalizar o Dockerfile. Se você deseja apenas usar o contêiner pré-compilado, consulte [Primeiros Passos](./getting-started).

## Clonar o Repositório

```bash
git clone https://github.com/f5xc-salesdemos/devcontainer.git
cd devcontainer
```

## Estrutura do Projeto

```text
.
├── Dockerfile                  # Multi-stage image build
├── docker-compose.yml          # Base compose — pulls pre-built image
├── docker-compose.build.yml    # Build override — use explicitly with -f
├── .devcontainer/              # VS Code Dev Container config
│   └── devcontainer.json
├── entrypoint.sh               # Container startup script
├── .env.example                # Example environment variables
├── docs/                       # Documentation site (Starlight)
├── claude-config/              # Claude Code configuration files
├── codex-config/               # Codex configuration
├── pi-config/                  # Pi configuration
└── omp-config/                 # Oh-My-Pi (omp) configuration
```

## Arquivo de Build

O arquivo `docker-compose.build.yml` adiciona suporte a build local. Ele **não** é mesclado automaticamente — você deve passá-lo explicitamente com as flags `-f`. Isso significa que `docker compose up -d` (ou `podman-compose up -d`) sempre puxa a imagem pré-compilada do GHCR, independentemente de você ter clonado o repositório ou não.

## Compilando Localmente

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
</Tabs>

Isso mescla o arquivo de build com o arquivo compose base, compila a imagem a partir do `Dockerfile` local e inicia o contêiner. A flag `--build` força uma recompilação mesmo que uma imagem em cache exista.

## Arquitetura de Build em Dois Estágios

O Dockerfile usa um build em dois estágios otimizado para cache de camadas do Docker:

### Estágio 1: `deps` (fundações estáveis, ~4,5 GB)

Essas camadas só são recompiladas quando um ARG de versão é atualizado ou os pacotes APT mudam. Em alterações típicas apenas de ferramentas, o BuildKit pula este estágio inteiro.

## Adicionando Ferramentas

Edite o `Dockerfile` e adicione sua ferramenta na seção apropriada. Recompile após adicionar:

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
</Tabs>

Ou no VS Code: **Dev Containers &rarr; Rebuild Container**.

### Pacotes APT

Adicione ao bloco `apt-get install` da seção 2:

```dockerfile
RUN apt-get update && apt-get install -y --no-install-recommends \
    your-package-here \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*
```

### Ferramentas npm

Adicione ao bloco `npm install -g` da seção 11:

```dockerfile
RUN npm install -g \
    your-npm-tool
```

### Ferramentas pip

Adicione ao bloco `pip install` da seção 12:

```dockerfile
RUN pip install --no-cache-dir --break-system-packages \
    your-python-tool
```

### Downloads binários

Adicione à seção 10 ou crie uma nova camada `RUN` com detecção de arquitetura:

```dockerfile
RUN ARCH=$(dpkg --print-architecture) \
    && curl -fsSL "https://example.com/tool-linux-${ARCH}.tar.gz" \
       | tar -xz -C /usr/local/bin tool
```

## Cache de Build

O CI usa **cache baseado em registro** — as camadas de build são armazenadas no GHCR (`ghcr.io/f5xc-salesdemos/devcontainer:cache-*`) em vez do cache do GitHub Actions. Isso evita o limite de 10 GB do cache do GHA, que causava recompilações completas frequentes para esta imagem multi-arquitetura de ~6 GB.

A arquitetura em dois estágios funciona com o cache de registro de forma que:

- **Atualizações de versão de ferramentas** (npm, pip, ferramentas binárias) — recompila apenas o estágio `final` (~5 min)
- **Alterações em arquivos de configuração** (`claude-config/`, `entrypoint.sh`) — recompila apenas as últimas camadas (~1 min)
- **Alterações nas fundações** (pacotes APT, versões Go/Rust/Java) — recompilação completa de ambos os estágios (~30 min)
