## 背景

2025 年 2 月，LayerX Security Research 揭露了 Anthropic Claude Desktop Extensions (DXT) 框架中一個嚴重的零點擊遠端程式碼執行漏洞。擴充功能在無沙箱環境下以完整系統權限運行，充當語言模型與本機作業系統之間的執行橋樑。

許多企業安全團隊現在要求 AI 程式碼工具僅在專用虛擬機器、容器或一次性環境中運行——而非直接在企業端點上執行。

**此 devcontainer 滿足該要求。**

## 隔離模型

```
+------------------------------------------------+
|  Host Machine (corporate endpoint)             |
|                                                |
|  - No AI tool binaries installed               |
|  - No AI tool extensions or plugins            |
|  - Only Docker is required                     |
|                                                |
|  +------------------------------------------+  |
|  |  Docker Container (isolated)             |  |
|  |                                          |  |
|  |  AI tools run here:                      |  |
|  |  - Claude Code CLI                       |  |
|  |  - Codex                                 |  |
|  |                                          |  |
|  |  No access to:                           |  |
|  |  x Host filesystem                       |  |
|  |  x Host keychain / credential store      |  |
|  |  x Host clipboard                        |  |
|  |  x Host network interfaces               |  |
|  |  x Other host applications               |  |
|  +------------------------------------------+  |
+------------------------------------------------+
```

## 安全控制措施

### 無主機磁碟區掛載

所有資料存放在 Docker 命名磁碟區中——而非綁定掛載的主機目錄。容器內部的入侵**無法**存取主機上的檔案。詳情請參閱[組態設定 — 資料持久化](./configuration#data-persistence)的磁碟區說明。

### 無共用憑證儲存區

容器無法存取 macOS 鑰匙圈、Windows 認證管理員、瀏覽器設定檔或主機 SSH 代理。SSH 金鑰在啟動時透過 base64 編碼的環境變數注入，且僅存在於容器內部。

### 無剪貼簿同步

Docker 容器不與主機共用剪貼簿狀態。資料無法透過複製/貼上洩漏。

### 網路隔離

容器透過以下兩種支援的驗證模式之一與外部世界通訊：

- **LiteLLM 代理模式** — 透過 Docker 的 NAT 向您的 LiteLLM 代理發送 HTTPS 出站請求。
- **OAuth 模式** — 透過 Docker 的 NAT 直接向 Anthropic 發送 HTTPS 出站請求。

容器無法存取 VPN 通道、企業網路介面或主機上的其他服務。

### 設計上的臨時性

整個環境可以在數秒內銷毀並重新建立。請將容器視為可拋棄的。

### 僅限 CLI 工具

此環境以**命令列介面**方式運行 AI 工具。沒有桌面應用程式、沒有瀏覽器擴充功能、沒有 DXT 框架。促使建立此隔離模型的漏洞不適用於容器內的 CLI 使用情境。

### 容器強化

`docker-compose.yml` 套用了多項縱深防禦措施：

- **能力丟棄** — 所有 Linux 能力均被丟棄（`cap_drop: ALL`），然後僅重新新增最小集合：`CHOWN`、`DAC_OVERRIDE`、`FOWNER`（用於入口點中的磁碟區權限修正）、`SETUID`/`SETGID`（用於 sudo）、`NET_RAW`（用於 ping/tcpdump）和 `NET_ADMIN`（用於 Wireshark 和 bettercap 等封包擷取工具）。
- **資源限制** — CPU（4 核心）、記憶體（8 GB）和 PID 數量（4096）透過 `mem_limit`、`cpus` 和 `pids_limit` 設定上限，以防止資源耗盡攻擊。
- **tmpfs 用於 `/tmp`** — `/tmp` 目錄掛載為 256 MB 的 tmpfs，將暫存檔案保留在記憶體中，防止它們持久化到容器的可寫入層。
- **預設啟用 TLS 驗證** — 未設定 `NODE_TLS_REJECT_UNAUTHORIZED`，因此 Node.js 預設會驗證 TLS 憑證。需要接受自簽憑證的使用者（例如用於內部代理）可以在 `.env` 檔案中新增 `NODE_TLS_REJECT_UNAUTHORIZED=0` 來選擇性啟用。
- **SSH 金鑰 umask** — SSH 私鑰在 `umask 077` 下寫入，因此檔案永遠不會是全域可讀的——即使在建立和 `chmod` 呼叫之間的短暫期間也不會。

### 安全性與滲透測試工具

容器包含約 80 個預先安裝的安全性與滲透測試工具，用於授權的測試情境。工具按類別組織：

| 類別 | 工具（範例） |
|----------|-----------------|
| 網路分析 | tshark、wireshark、masscan、hping3、bettercap（amd64）、netdiscover |
| Web 掃描器 | nikto、sqlmap、dirb、whatweb、sslscan、OWASP ZAP、dalfox、feroxbuster |
| 密碼與驗證 | hydra、john、hashcat、medusa、ncrack |
| 逆向工程 | radare2、Ghidra、gdb、binwalk、strace、ltrace |
| 偵察 | subfinder、amass、httpx、nuclei、gau、waybackurls、recon-ng、spiderfoot |
| 模糊測試與列舉 | ffuf、gobuster、SecLists |
| 供應鏈與密鑰 | trufflehog、grype、syft、gitleaks、trivy |
| 滲透框架 | Metasploit（amd64）、searchsploit（ExploitDB） |

部分工具（bettercap、Metasploit）由於上游套件限制，僅在 amd64 上可用。封包擷取工具（tshark、bettercap）需要 `NET_ADMIN` 能力，該能力已包含在預設的 `docker-compose.yml` 中。

如需完整列表，請在容器內執行 `claude-self-test`（第 7 節會檢查所有安全工具）。

## 最佳實踐

1. **請勿在主機上安裝 AI 程式碼工具。** 請改用此容器。
2. **請勿新增綁定掛載**至 `docker-compose.yml`。命名磁碟區組態是刻意為之的。
3. **定期輪換**容器內使用的 SSH 金鑰。
4. **分享前請審查 `.env`。** 其中包含 API 金鑰，且可能包含 SSH 私鑰。
5. **定期更新**以取得安全性修補程式：
