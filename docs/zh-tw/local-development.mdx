本頁面適用於想要複製儲存庫、在本機建置映像或自訂 Dockerfile 的開發者。如果您只是想使用預建容器，請參閱[快速入門](./getting-started)。

## 複製儲存庫

```bash
git clone https://github.com/f5xc-salesdemos/devcontainer.git
cd devcontainer
```

## 專案結構

```text
.
├── Dockerfile                  # Multi-stage image build
├── docker-compose.yml          # Base compose — pulls pre-built image
├── docker-compose.build.yml    # Build override — use explicitly with -f
├── .devcontainer/              # VS Code Dev Container config
│   └── devcontainer.json
├── entrypoint.sh               # Container startup script
├── .env.example                # Example environment variables
├── docs/                       # Documentation site (Starlight)
├── claude-config/              # Claude Code configuration files
├── codex-config/               # Codex configuration
├── pi-config/                  # Pi configuration
└── omp-config/                 # Oh-My-Pi (omp) configuration
```

## 建置檔案

`docker-compose.build.yml` 檔案新增了本機建置支援。它**不會**自動合併 — 您必須使用 `-f` 旗標明確傳入。這代表 `docker compose up -d`（或 `podman-compose up -d`）無論您是否已複製儲存庫，都會從 GHCR 拉取預建映像。

<Tabs syncKey="runtime">
<TabItem label="Docker">

您可以驗證單純執行 `docker compose up` 只會使用預建映像（沒有 `build:` 上下文）：

```bash
docker compose config
```

與明確的建置組態進行比較：

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml config
```

</TabItem>
<TabItem label="Podman">

您可以驗證單純執行 `podman-compose up` 只會使用預建映像（沒有 `build:` 上下文）：

```bash
podman-compose config
```

與明確的建置組態進行比較：

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml config
```

</TabItem>
</Tabs>

## 本機建置

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
</Tabs>

這會將建置檔案與基礎 compose 檔案合併，從本機 `Dockerfile` 建置映像，並啟動容器。`--build` 旗標會強制重新建置，即使已存在快取映像。

## 兩階段建置架構

Dockerfile 使用針對 Docker 層快取最佳化的兩階段建置：

### 階段 1：`deps`（穩定基礎，約 4.5 GB）

這些層只會在版本 ARG 變更或 APT 套件變動時重新建置。在一般僅工具變更的情況下，BuildKit 會跳過整個階段。

| 區段 | 內容 |
|---------|----------|
| 1. APT 儲存庫 | NodeSource、deadsnakes、HashiCorp、GitHub、Docker、Microsoft、Google Cloud、Dart SDK |
| 2. APT 套件 | 系統工具、Node.js、Python、Java、Terraform、GitHub CLI、Docker 引擎、Azure CLI、PowerShell、地區設定 |
| 2b. 安全性 APT 套件 | 網路分析、網頁掃描器、密碼工具、逆向工程、數位鑑識（約 80 個套件） |
| 3. Python 初始化 | 符號連結 + 透過 get-pip.py 安裝 pip |
| 4. Go | 官方 tarball（最新穩定版，建置時解析） |
| 5. Rust | rustup（系統範圍，最新穩定版） |

## 新增工具

編輯 `Dockerfile` 並將您的工具新增至適當的區段。新增後重新建置：

<Tabs syncKey="runtime">
<TabItem label="Docker">

```bash
docker compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
<TabItem label="Podman">

```bash
podman-compose -f docker-compose.yml -f docker-compose.build.yml up -d --build
```

</TabItem>
</Tabs>

或在 VS Code 中：**Dev Containers &rarr; Rebuild Container**。

### APT 套件

新增至區段 2 的 `apt-get install` 區塊：

```dockerfile
RUN apt-get update && apt-get install -y --no-install-recommends \
    your-package-here \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*
```

### npm 工具

新增至區段 11 的 `npm install -g` 區塊：

```dockerfile
RUN npm install -g \
    your-npm-tool
```

### pip 工具

新增至區段 12 的 `pip install` 區塊：

```dockerfile
RUN pip install --no-cache-dir --break-system-packages \
    your-python-tool
```

### 二進位檔下載

新增至區段 10 或建立一個帶有架構偵測的新 `RUN` 層：

```dockerfile
RUN ARCH=$(dpkg --print-architecture) \
    && curl -fsSL "https://example.com/tool-linux-${ARCH}.tar.gz" \
       | tar -xz -C /usr/local/bin tool
```

## 建置快取

CI 使用**基於登錄檔的快取** — 建置層儲存在 GHCR（`ghcr.io/f5xc-salesdemos/devcontainer:cache-*`）中，而非 GitHub Actions 快取。這可以避免 10 GB GHA 快取限制，該限制對這個約 6 GB 的多架構映像造成頻繁的完整重新建置。

兩階段架構與登錄檔快取搭配運作，使得：

- **工具版本升級**（npm、pip、二進位工具） — 僅重新建置 `final` 階段（約 5 分鐘）
- **組態檔案變更**（`claude-config/`、`entrypoint.sh`） — 僅重新建置最後幾層（約 1 分鐘）
- **基礎變更**（APT 套件、Go/Rust/Java 版本） — 兩個階段完整重新建置（約 30 分鐘）
